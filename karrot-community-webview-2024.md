# 아니, 여기도 웹뷰였어요? | 2024 당근 테크 밋업

> 발표자: 로토 (당근 동네생활 팀)
> 출처: https://www.youtube.com/watch?v=4UD4EB00AME

## 발표 개요

당근의 **동네생활** 서비스(글 쓰기, 글 상세, 피드)는 원래 100% 네이티브로 구현되어 있었다. 동네생활 팀은 이를 웹뷰로 **역방향 전환**(네이티브 → 웹뷰)을 진행했다. 보통은 "웹뷰가 느리니까 네이티브로 가자"가 일반적인 흐름이지만, 동네생활 팀은 정반대 결정을 내렸다.

이 문서는 다음 흐름으로 발표 내용을 정리한다.

1. **왜 웹뷰로 가야 했는가** — 문제 정의
2. **느리지 않은 웹뷰를 만들기 위한 의사결정** — CSR의 한계 → SSR → 스트리밍 SSR
3. **전환 이후에 생긴 새로운 문제들** — 운영, 플랫폼 차이, 배포 전파, 오프라인

## 1. 왜 웹뷰로 갔는가: 문제 정의

### 출발점의 문제 정의

> "UI를 좀 더 때려부수고, 새로운 시도를 해보고 싶다."

- 동네생활 전체가 네이티브로 잘 동작하고 있었지만, **빠른 실험과 잦은 UI 변경**이 어렵다는 본질적 한계가 있었다.
- AB 테스트를 통한 가설 검증 사이클이 느리다는 것이 핵심 페인포인트.

### 의사결정의 흐름

1. 먼저 **글 쓰기 / 글 상세** 화면만 웹뷰로 전환 (작년에 완료)
2. 전환 후 AB 테스트를 본격적으로 돌려보니 **네이티브 대비 훨씬 많은 실험과 성과** 확보
3. 진입점인 **피드까지 웹뷰로 전환** (2024년 5월 완료)

> **패턴**: 큰 결정을 한 번에 내리지 않고, 가장 변화가 잦은 영역(글쓰기·상세)부터 작게 전환해 효과를 데이터로 증명한 뒤, 진입점까지 확장. 즉 **"위험이 적은 곳에서 효과를 입증하고 핵심으로 확장"** 하는 점진적 마이그레이션.

## 2. "느리지 않게 해 달라"는 요구사항: CSR의 한계 분석

### 핵심 제약 조건

> "비교 대상이 네이티브이기 때문에, 최대한 빠르게."

이 한 문장이 모든 기술 선택의 기준이 된다. 일반적인 웹 서비스라면 CSR로도 충분하지만, **사용자가 직전까지 네이티브를 쓰던 화면**에서 웹뷰로 갔을 때의 체감 속도 저하는 곧 제품의 후퇴를 의미한다.

### CSR의 두 가지 구조적 약점

#### (1) 배포 플랫폼 문제 — CDN 리전

- 일반적인 CSR은 배포 시 새 번들을 받아오는데, 네트워크가 안 좋거나 번들이 크면 화면이 빈 상태로 멈춰 있는 White Screen이 발생.
- 더 큰 문제는 **CDN 리전**: 클라우드플레어 같은 도구를 그냥 쓰면 한국 사용자가 미국 리전을 다녀오는 일이 생김. 첫 진입 시 10초씩 화면이 멈추는 케이스 발생.
- 해결책 한 가지는 직접 AWS S3 + CloudFront로 파이프라인을 짜는 것이지만, **운영 비용이 너무 비싸짐**.

#### (2) API 호출 시점 문제 — Cross-domain Preflight

- CSR은 번들이 모두 로딩된 뒤에야 브라우저에서 API를 호출.
- CSR은 보통 서버리스 환경에서 동작 → API 도메인과 정적 자원 도메인이 다름 → **CORS Preflight 요청 필수**.
- 한 번의 Preflight가 100ms 정도 지연을 만들고, API를 4~5개 호출하면 이 지연이 누적되어 무시할 수 없는 수준이 됨.
- 거기에 **모바일 사용 환경의 네트워크 변동성**(지하철, 엘리베이터 등) 까지 더해지면 CSR의 체감 속도는 크게 흔들림.

> **패턴**: 단순히 "느리다"가 아니라, *어떤 단계에서, 어떤 메커니즘으로, 얼마나 느려지는가*를 구체적인 숫자(100ms)와 사용 시나리오(엘리베이터, 지하철)로 분해. 추상적인 성능 문제를 구체적이고 측정 가능한 형태로 환원.

## 3. 해결: SSR 직접 구축

### 의사결정

- **Next.js / Remix 같은 풀 프레임워크 대신 직접 구축**.
- 이유:
  - Next.js는 Vercel 친화적인데, 사내 방향이 Vercel을 쓰지 않는 쪽.
  - Remix는 SSR 자체를 처음 도입하는 팀에 사내 도구와의 호환이 충분히 맞지 않음.
- 대안: **Vite + React Server**로 SSR 파이프라인을 직접 구성.

### CSR vs SSR 동작 비교

| 단계 | CSR | SSR |
|---|---|---|
| index.html 응답 | 빈 DOM | 렌더링된 HTML |
| API 호출 주체 | 브라우저 | 노드 서버 |
| 첫 화면이 사용자에게 보이는 시점 | JS 번들 로드 + API 호출 완료 후 | HTML 도착 시 |
| Preflight 영향 | 있음 | 서버-서버 통신이라 없음 |
| 사용자 네트워크 환경 영향 | 큼 | 작음 |

> **패턴**: SSR을 도입하는 결정의 근거는 "트렌드"나 "SEO" 같은 일반론이 아니라, **CSR의 두 가지 약점(CDN 리전 + Preflight + 사용자 네트워크 변동성)을 직접 지운다**는 매우 구체적인 메커니즘 차이. *왜 이 도구가 우리의 문제를 푸는가* 를 직접 매핑.

## 4. SSR의 단점과 해결: Streaming SSR

### 새로운 문제 정의

- SSR은 **모든 API 응답이 끝나야 HTML을 응답**으로 보낸다.
- API가 느리면 사용자에게는 오히려 CSR보다 화면이 늦게 뜨는 것처럼 보일 수 있다.
- 발표자의 표현: *"과거에 JSP 같은 데서 API 느려서 CSR로 도망간 기억이 있는데, 거꾸로 돌아온 셈."*

### Streaming SSR 도입

- 화면을 **한 번에 그리지 않고, 단계적으로 그려서 내려보낸다**.
- API 응답과 무관한 부분(GNB, 상단 바 등)은 **Shell** 로 정의해 먼저 보낸다.
- API 응답이 필요한 부분은 **`<Suspense>`로 감싸서**, 데이터가 준비되는 대로 청크 단위로 내려간다.

### 구현 도구

- React의 `renderToPipeableStream`
- `onShellReady` 콜백에서 `pipe(res)` 호출 시 스트리밍 시작
- `<Suspense fallback={...}>` 로 비동기 영역을 정의

### 사용자 체감

- 진입 즉시 헤더·레이아웃부터 보임 → "들어가자마자 무언가 뜬다"는 인상.
- 본문은 fallback(스켈레톤 등)으로 자연스럽게 채워짐.
- *"화면이 없다가 튀어나오는 것"* 과 *"뼈대라도 보이고 채워지는 것"* 의 체감 차이가 결정적.

> **패턴**: SSR이라는 해결책이 만들어낸 새로운 문제(올오어낫싱 렌더링)를, *해결책을 버리지 않고* React가 제공하는 `<Suspense>` 라는 동일 생태계의 기본기로 해결. 문제→해결→새 문제→점진적 보강의 자연스러운 흐름.

## 5. 전환 이후 마주한 운영 문제들

웹뷰로 전환한 뒤에는 **CSR/SSR이 멋있어 보이는 부분 너머의** 문제들이 줄줄이 따라왔다. 발표자는 *"SSR을 무작정 추천 못 하는 이유"* 라며 솔직하게 공유.

### 5-1. iOS Safari 키보드 — "흑마술"

- iOS Safari는 포커스된 input이 키보드 아래에 있으면 **강제로 화면을 위로 밀어버린다**. `preventDefault`로도 못 막음.
- 이 동작 때문에 특정 UI 조작이 불가능해지는 케이스가 발생.
- 해결: 트리키한 우회법으로 처리했지만, *"언제 막힐지 모른다"* 는 본질적 불안 존재.

> **패턴**: 플랫폼이 강제하는 동작은 정공법으로 못 막는 경우가 있다. 흑마술적 해결을 솔직히 인정하고, **언제 깨질지 모른다는 리스크를 명시적으로 관리**.

### 5-2. SSR 도입 = 운영 업무 추가

- CSR은 별도 서버가 없어 프론트엔드 엔지니어의 운영 부담이 적음(Sentry 정도).
- SSR은 **노드 서버를 운영**해야 함 → CPU/메모리/응답 시간/트래픽 스파이크 모니터링이 프론트엔드 엔지니어의 업무가 된다.
- *"SSR이 좋다는 걸 알면서도, 다른 팀에게 무작정 권하기 어려운 이유."*

> **패턴**: 기술 선택은 **그 기술을 운영할 팀의 역량과 부담**까지 포함해서 평가해야 한다. 좋은 기술을 도입했다고 끝이 아니다.

### 5-3. 안드로이드 / iOS 디자인·로깅 차이

- 두 OS는 네이티브 디자인이 다르므로, 웹뷰가 둘 다 자연스럽게 동작하려면 합의가 필요.
- 글쓰기·상세는 **스펙 정리**로 해결 ("이 부분은 안드 기준, 이 부분은 iOS 기준").
- **피드는 데이터 로깅 기준이 OS마다 달라서**, 전환 가부를 결정짓는 데이터가 어긋남.
- 결과: **개발 시간보다 데이터를 맞추는 시간이 더 오래 걸림**.

> **패턴**: 마이그레이션은 *기능 동일성*만 맞추면 끝이 아니다. **로그·메트릭·실험 지표 기준의 동일성**까지 맞춰야 비로소 비교 가능한 출시가 된다. 이 비용을 미리 예상해두는 것이 중요.

### 5-4. 배포가 사용자에게 전파되지 않는 문제

#### 문제 정의
- 웹뷰니까 배포만 하면 모든 사용자가 자동으로 최신 버전을 쓸 거라고 **나이브하게 생각**.
- 글쓰기/상세는 매번 새로 띄우는 화면이라 자연스럽게 갱신.
- 그러나 **iOS 사용자**는 앱을 끄지 않고 백그라운드에 두는 사용 패턴이 일반적. → 배포해도 전파되지 않음.

#### 해결 방법

1. 노드 서버가 응답에 **현재 빌드 버전**을 포함해서 내려준다.
2. 앱 측 글로벌 디스(글로벌 데이터 스토어)에 **렌더링 시점의 앱 버전**을 보관.
3. `visibilitychange` 이벤트를 이용해 **앱이 백그라운드에서 돌아오는 순간**을 감지.
4. 그 시점에 서버 버전과 클라이언트 버전을 비교 → 다르면 **사용자에게 새로고침을 유도**(강제 X, 토스트로 권유).

> **패턴**: "당연히 그럴 것"이라고 가정하지 않고, **실제 사용 패턴(iOS 백그라운드 유지)에 맞춰 가정을 검증**. 그리고 강제 갱신이 아닌 **사용자 친화적 권유** 방식 선택.

### 5-5. 새로고침 → 오프라인 케이스

새로고침 유도까지 만들었더니, 또 다른 문제가 생긴다.

#### 문제 정의
- 사용자가 새로고침을 누른 시점에 **오프라인일 수 있다**(엘리베이터, 지하철).
- 오프라인에서 `location.reload()` 가 호출되면 **웹뷰가 그냥 먹통**이 됨.

#### 해결 방법
- `window.online` / `window.offline` 이벤트를 듣는 **커스텀 훅** 작성.
- 새로고침 시점에 온라인이면 → 즉시 reload.
- 오프라인이면 → *"네트워크가 불안정합니다"* 토스트로 안내.
- 오프라인 상태에서 새로고침이 시도된 횟수를 **별도 로깅** (펜드리 큐 사용 추정 — 온라인 복귀 시 전송).

> **패턴**: 해결책이 만들어낸 *그다음 엣지 케이스*까지 한 단계 더 깊이 본다. **"이 해결책이 깨지는 시나리오는?"** 을 한 번 더 물어보는 습관.

### 5-6. 사파리 호환성 — 디테일의 함정

- *"iOS 14 사파리에서 `flex` 의 `gap` 이 안 된다."* 그런데 **`grid` 의 `gap` 은 된다.**
- 이런 디테일이 모르고 지나치면 사용자 일부에서 깨진 UI가 노출됨.
- 보수적으로 지원해야 하는 OS 버전이 있다면 이런 호환성 매트릭스를 **미리 알고 있어야** 한다.

## 6. 결론: 그럼에도 웹뷰의 가치

발표자가 마무리에 정리한 웹뷰 도입의 정당성:

1. **빠른 개발 주기와 배포 사이클** — 잦은 AB 테스트로 가설 검증 속도를 극대화.
2. **사용자에게 더 가치 있는 제품을 빠르게 전달**.
3. **대부분의 케이스에서 웹뷰만으로도 충분히 만족스러운 성능과 UX 제공**이 가능.

> 단, 그 결정에는 **CSR의 한계 분석 → SSR → 스트리밍 SSR → 운영 부담 수용 → OS별 차이 흡수 → 배포 전파 + 오프라인 처리** 까지의 모든 과제를 풀어내겠다는 각오가 함께여야 한다.

## 부록: 발표에서 드러난 문제 해결 패턴 정리

이 발표를 관통하는 사고 방식을 한 번 더 추출.

| 단계 | 패턴 | 적용 사례 |
|---|---|---|
| **문제 정의** | 추상적 불만(느리다)을 구체적 메커니즘(Preflight 100ms × N개 + 모바일 네트워크 변동성)으로 분해 | CSR 한계 분석 |
| **해결 선택** | 트렌드가 아니라 *바로 그 메커니즘을 지우는가* 를 기준으로 도구 선택 | SSR 직접 구축 (Next/Remix 거부) |
| **점진적 적용** | 변화가 잦고 위험이 적은 영역부터 작게 전환, 데이터로 증명 후 핵심 영역 확장 | 글쓰기/상세 → 피드 |
| **새 문제 인정** | 해결책이 만든 새 문제(올오어낫싱 SSR)를 동일 생태계의 기본기로 다시 해결 | Streaming SSR + Suspense |
| **운영 비용 명시화** | 좋은 기술의 도입 = 그 기술을 운영할 부담의 도입. 이를 솔직히 공유 | SSR 운영 부담, "다른 팀엔 권하기 어렵다" |
| **사용 패턴 검증** | "당연히 그럴 것" 같은 가정을 실제 OS·사용자 행동으로 검증 | iOS 백그라운드 유지로 인한 배포 전파 실패 |
| **엣지 케이스 한 단계 더** | 해결책이 깨지는 시나리오를 한 번 더 묻기 | 새로고침 → 오프라인 처리 |
| **강제보다 유도** | 강제 갱신 대신 사용자 친화적 안내 선택 | 버전 다를 때 토스트로 권유 |
| **흑마술 솔직히 인정** | 정공법이 안 되는 영역에선 우회를 인정하고 리스크를 관리 | iOS Safari 키보드 처리 |
