# 좋은 개발자 핸드북 — 컨퍼런스 발표 3편에서 뽑아낸 룰·패턴·액션

세 발표(토스 SLASH 22 / 2024 당근 테크 밋업 ×2)의 분석 산출물을, 흩어진 형태가 아니라 한 호흡에 읽을 수 있는 핸드북으로 묶었다. 새 주장은 더하지 않았으며, 산출물의 발췌·재구성·링크로만 구성된다.

## Part 0. 이 문서가 만들어진 과정

세 컨퍼런스 발표 정리 문서를 입력으로, 도메인을 걷어낸 보편 룰까지 단계적으로 도출하는 7단계 절차를 거쳤다 — (1) 입력 자료 점검, (2) 문서별 정규화 분석, (3) 패턴 수렴(cross-document), (4) 추상 룰 도출, (5)~(6) 문제 정의·해결 패턴 카탈로그, (7) 액션 플랜 도출. 각 단계의 입력은 직전 단계의 산출물이며, 원본 발표를 직접 인용하지 않은 경우 단계 점프는 허용하지 않았다.

## Part 1. 추상 룰 (요약 + 우선순위)

**Rule 1.** 추상적 통증은 현재 상태를 펼쳐 보인 뒤 메커니즘과 측정 가능한 항으로 분해한다. *분해된 통증만이 도구 채택의 매핑 기준이 된다.*

**Rule 2.** 익숙해진 가정과 표준은 결정의 입구마다 다시 한 번 의심한다. *가정 위에 쌓인 결정은 가정이 깨지는 순간 한 번에 무너진다 — 이번 결정의 핵심 가정 한두 개만 명시적으로 의심.*

**Rule 3.** 풀 수 없는 형태로 정의된 문제를, 설계 가능한 형태로 다시 정의한다. *"예측" → "대응", "합의 가속" → "합의 데이터화"처럼 동사를 바꿀 때만 진척이 생긴다.*

**Rule 4.** 모호함이 큰 결정의 입구에는, 한 문장의 정렬축을 먼저 만든다. *비교 대상이나 매체 명시 같은 구체성이 들어가야 정렬 효과가 생긴다.*

**Rule 5.** 수명이 다른 책임은 같은 단위에 묶지 않고 다른 단위로 흩는다. *분리 자체가 목적이 되어선 안 된다 — 분리할 때마다 "왜 분리하는가"를 묻는다.*

**Rule 6.** 사용자가 보는 면(인터페이스·명세)을 구현보다 먼저 결정한다. *사용 코드부터 적어 보면 인터페이스의 어색함이 먼저 드러난다.*

**Rule 7.** 도구는 트렌드가 아니라 "우리 통증의 메커니즘을 직접 지우는가"로 채택한다. *매핑 1줄을 코드에 박아 두면 1년 뒤에도 의도를 복원할 수 있다.*

**Rule 8.** 부품의 경계는 서로의 존재를 모르도록 독립적으로 두되, 합쳐질 통합 지점은 미리 설계한다. *통합 지점이 사후에 짜이면 양쪽 부품이 모두 오염된다.*

**Rule 9.** 해결책으로 끝내지 말고, 그 해결책이 만든 다음 라운드의 통증까지 같은 사고에 포함시킨다. *"기능 동일"이 아니라 "메트릭·운영·다음 엣지까지 동일"이 끝점.*

**Rule 10.** 반복되는 사람 작업은 추상화·도구화 부재의 신호로 읽는다. *회의가 반복된다는 것은 그 회의에서 다루던 정보가 데이터로 표현될 수 있다는 직접 신호.*

### 룰 우선순위

가장 먼저 몸에 들이는 1순위는 **Rule 1**(다른 모든 룰의 시작점), 2순위는 **Rule 9**(가장 자주 어기는 룰), 3순위는 **Rule 6**(일상 코드 단위에서 가장 자주 적용 가능). 이 셋을 들인 뒤 Rule 2·4·5·7·8·10을 결정의 입구마다 점검 항목으로, Rule 3은 큰 프레임 결정 시점에만 의식적으로 꺼내 쓴다.

## Part 2. 문제를 올바르게 정의하는 법

여섯 패턴을 "현재 가시화 → 메커니즘 분해 → 가정 의심 → 본질 질문으로 좁히기 → 풀 수 있는 형태로 프레임 전환 → 다음 라운드까지 포함"의 순서로 적용한다.

### 패턴 1. Visibility-First — 통증을 말하기 전에 현재 절차와 자기 코드를 펼쳐 보인다
- **자가 점검**: 이 통증을 그림이나 표로 그릴 수 있는가? 본인이 직접 만든 실패 사례를 한 개 이상 가져올 수 있는가?

### 패턴 2. Mechanism Decomposition — 추상 통증을 단계 × 메커니즘 × 측정 항으로 분해한다
- **자가 점검**: 이 통증의 항목별 크기를 숫자로 적을 수 있는가? 가장 큰 항을 직접 지우는 도구를 1개 이상 떠올릴 수 있는가?

### 패턴 3. Assumption Audit — 익숙한 가정·표준의 적합성을 결정 입구마다 의심한다
- **자가 점검**: 이 결정의 핵심 가정 한두 개를 종이에 적을 수 있는가? 그 가정이 깨질 시나리오를 한 개 이상 떠올릴 수 있는가? 익숙한 표준이 이 매체에서도 여전히 가장 단순한가?

### 패턴 4. Essential Question Framing — 분해 결과를 한 문장의 본질 질문으로 좁힌다
- **자가 점검**: 이번 통증을 한 문장의 질문으로 적을 수 있는가? 그 질문이 추상적이지 않고 구체적인가?

### 패턴 5. Reframing — 풀 수 없는 형태의 문제를 설계 가능한 형태로 다시 정의한다
- **자가 점검**: 지금 목표가 본질적으로 풀 수 있는 형태인가? 같은 통증을 다른 동사로 다시 정의할 수 있는가?

### 패턴 6. Next-Round Inclusion — 해결책이 만든 다음 라운드의 통증을 같은 사이클에 포함시킨다
- **자가 점검**: 이 해결책이 만들 새 비용 한 가지를 적을 수 있는가? 깨지는 시나리오 한 가지를 미리 적었는가? 비교에 필요한 메트릭이 새 환경에서도 동일하게 잡히는가?

## Part 3. 지혜롭게 해결하는 법

"지혜롭다"의 조작적 정의: ① 더 작은 결정으로 더 많은 변화를 흡수 ② 자의적 판단을 측정 가능한 기준으로 ③ 단기 통증을 감수하고 시스템 본질을 단순화 ④ 협업·인지 비용을 코드 구조로 흡수.

### 패턴 1. Lifetime-Based Separation
- **핵심 메커니즘**: 한 단위 안에서 변하는 속도가 다른 책임을 분리해, 짧은 수명의 변경이 긴 수명을 흔들지 않게 한다 (① + ④).
- **흔한 오용**: 분리 자체가 목적이 되어 모든 것을 잘게 나눈 결과 통신 비용이 분리 이익을 초과.

### 패턴 2. Interface-Before-Implementation
- **핵심 메커니즘**: 사용 코드부터 적어 인터페이스를 먼저 굳혀, 구현 사정이 사용자 면을 오염시키는 것을 막는다 (② + ④).
- **흔한 오용**: 진짜 요구사항을 모르는 영역에서 인터페이스를 너무 일찍 굳혀 잘못된 추상에 갇힘.

### 패턴 3. Mechanism-Mapped Adoption
- **핵심 메커니즘**: 도구 채택을 "트렌드"·"의리"의 자의에서 "어느 메커니즘을 직접 지우는가"의 측정 가능 기준으로 바꿈 (②).
- **흔한 오용**: 매핑을 거창하게 만들어 도구 채택 자체가 무거워짐 — 매핑은 1줄이면 충분.

### 패턴 4. Single-Sentence Alignment Axis
- **핵심 메커니즘**: 후속 수십 개 결정의 모호함을 한 문장이 지운다 — 비교 대상·매체 명시 같은 구체성 필수 (② + ④).
- **흔한 오용**: 추상적인 정렬축은 효과 0. 만들어 두고 후속 결정에서 안 꺼내는 상태도 흔함.

### 패턴 5. Decoupled Composition with Pre-Designed Integration
- **핵심 메커니즘**: 부품 독립이 변화를 부품 단위로 흡수, 통합 지점이 사전 설계되어 있으면 합 이상의 가치가 자연 발생 (①).
- **흔한 오용**: 통합 지점이 사후에 끼워 맞춰져 양쪽 부품 인터페이스가 모두 오염.

### 패턴 6. Single Source of Truth with Auto-generation
- **핵심 메커니즘**: 회의·합의·수동 절차로 흩어진 정보를 한 데이터 원천으로 모으고, 그 원천에서 산출물을 자동 생성. 한 번의 도구화로 무한 반복 비용을 0에 가깝게 (④ + ①).
- **흔한 오용**: 빈도가 낮거나 매번 미묘하게 다른 영역에 적용해 도구화 비용이 가치를 초과.

### 패턴 7. External-as-Learning-Tool
- **핵심 메커니즘**: 자체 구축의 매혹을 보류하고 외부 솔루션을 학습 도구로 충분히 운영, 진짜 요구사항이 보일 때 도메인 특화 확장만 자체 구축 (③).
- **흔한 오용**: "충분히 써본"의 정의가 모호해 사실상 한 번 써보고 자체 구축으로 넘어감. 자체 구축이 외부 클론으로 끝나는 경우.

### 패턴 8. Next-Round-In-Same-Thought
- **핵심 메커니즘**: 단기 통증(운영 부담, 한계 솔직, 메트릭 동일성 비용)을 같은 사고에 포함하고 사후 비용을 미리 코드/구조로 흡수 (③ + ④).
- **흔한 오용**: 다음 라운드를 한 번에 다 풀려 해 결정이 무한 미뤄짐. 또는 메모로만 남고 후속 추적 없음.

## Part 4. 오늘부터 할 수 있는 일

세 산출물(Part 1·2·3)을 일상 행위로 번역한 17개 액션. 각 액션의 "연결되는 룰/패턴"이 Part 1/2/3과의 링크 역할을 한다.

### 0. 사용법

- **한 번에 다 적용하지 말고, 한 사이클(1주)에 1~2개만 새로 시도**한다. 룰 우선순위(1·9·6)를 따라 들어가는 것이 가장 빠르다.
- 모든 액션에는 **외부 산출물**(노트·PR 본문·코드 주석)이 있어야 한다. 머릿속에서만 한 액션은 안 한 것과 같다.
- 한 주를 끝낼 때 **W1만 빠뜨리지 않는다**. W1은 다른 액션의 누적을 점검하는 메타 액션이다.
- 안티 액션(§6)은 시작 시점이 아니라 **결정 직전의 STOP 사인**으로 쓴다.

### 1. 매일 단위 (Daily)

**액션 D1.** "느리다·복잡하다·어렵다" 통증을 만나면 즉시 메커니즘으로 분해해 종이에 적기. 연결: **Rule 1**, Part 2 패턴 2. → 매일 1회.

**액션 D2.** 새 함수·컴포넌트를 만들기 전, "사용 코드 3줄"을 IDE에 먼저 적기. 연결: **Rule 6**, Part 3 패턴 2. → 매일 2~3회.

**액션 D3.** 단위를 분리하려는 순간 "왜 분리하는가"를 한 줄로 적기. 연결: **Rule 5**, 안티 액션 A1. → 매일 1~2회.

### 2. 매 PR / 매 과제 단위

**액션 P1.** PR 본문 첫 줄에 "통증의 메커니즘"을 1줄 적기 — `통증: X가 (단계/메커니즘/얼마나) → 이 PR이 N번째 항을 직접 지움`. 연결: **Rule 1, 7**, Part 3 패턴 3. → 매 PR.

**액션 P2.** PR 머지 직전 "이 변경이 만든 다음 라운드 통증 1개"를 본문 끝에 적기. 연결: **Rule 9**, Part 2 패턴 6, Part 3 패턴 8. → 매 PR.

**액션 P3.** PR 자기 코멘트에 "이번에 적당히 한 부분 1개"를 직접 짚기. 연결: **Rule 1**, 안티 액션 A1. → 매 PR.

**액션 P4.** 새 의존성·도구 도입 PR에는 "메커니즘 매핑 1줄"을 코드 주석으로 박기. 연결: **Rule 7**, Part 3 패턴 3. → 매 도구 추가 시.

### 3. 매주 단위 (Weekly)

**액션 W1.** 주말에 "어겼던 룰 1개 + 통증 분해 노트 1편 다듬기" 회고. 연결: **Rule 우선순위(1·9·6)**, Rule 10. → 주 1회 (가장 빠뜨리지 말 것).

**액션 W2.** 한 주의 익숙한 가정 1개를 종이에 적고 "깨질 시나리오 1개"를 떠올리기. 연결: **Rule 2**, Part 2 패턴 3. → 주 1회.

### 4. 매월 단위 (Monthly)

**액션 M1.** 한 달 분 통증 분해 노트를 모아 "반복 메커니즘 1개" 패턴화. 연결: **Rule 10**, Part 2 흐름도. → 월 1회.

**액션 M2.** 자신의 현재 큰 작업에 대한 "한 문장 정렬축" 다듬기. 연결: **Rule 4**, Part 3 패턴 4. → 월 1회.

### 5. 우테코 맥락 특화 액션

**액션 U1.** 페어 시작 전 "사용 코드 3줄"을 둘이 합의해서 먼저 적기. 연결: **Rule 6**, Part 3 패턴 2. *왜 우테코에서 효과적인가*: 페어가 인터페이스 우선을 강제하는 가장 자연스러운 시점이며, 합의된 3줄이 페어 변경 시 인계 자료가 된다. → 매 페어 시작 시.

**액션 U2.** 미션 회고 양식에 "메커니즘 분해 1개 + 다음 라운드 1개" 칸을 의무로 추가. 연결: **Rule 1, 9**, Part 2 패턴 2 + 6. *왜 우테코에서*: 회고가 우테코 크루의 가장 일관된 산출물 — 칸을 박으면 매 미션마다 룰 1·9가 강제 적용된다. → 매 미션 종료 시.

**액션 U3.** 크루 코드 리뷰에 "통증 매핑 질문 1개" 코멘트 룰. 연결: **Rule 7**, Part 3 패턴 3. *왜 우테코에서*: 크루 리뷰는 횡적 학습의 자리 — 한 사람의 리뷰가 다른 크루의 룰 7 체화로 전파된다. → 도구 도입 PR 리뷰마다.

**액션 U4.** 페어 종료 시 "오늘의 안티 액션 1개"를 둘이 짚어 노트에 남기기. 연결: 안티 액션 §6. *왜 우테코에서*: 페어가 자의적 판단을 외부 시선으로 보게 해 주는 가장 강력한 장치. → 매 페어 종료 시.

### 6. 안티 액션 (피해야 할 STOP 사인)

- **A1.** "코드가 커져서 분리한다"는 PR 사유 금지 — 이름이 한 줄로 안 나오면 분리 취소.
- **A2.** "X가 좋대서 도입했다" 도구 채택 금지 — 메커니즘 매핑(P4) 없는 의존성 추가는 PR 거절 사유.
- **A3.** "기능은 됐어요"로 종료 선언 금지 — 메트릭 동일성 + 운영 부담 + 다음 라운드 통증(P2)을 적기 전엔 "끝"이 아니다.
- **A4.** "이건 당연히 X일 거예요"의 가정 위에서 결정 금지 — 가정이 깨질 시나리오 1개(W2)를 먼저 적는다.
- **A5.** 같은 회의·합의를 도구화 없이 두 번 이상 반복 금지 — 두 번째 회의가 잡히는 순간 "이걸 데이터로 표현할 수 있는가?" 한 줄을 적고, 답이 "예"면 다음 회의 대신 도구화 PR을 연다.
