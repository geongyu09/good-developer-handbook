# 프레임워크부터 플랫폼까지: 당근 웹뷰 플랫폼 (2024 당근 테크 밋업)

> 발표자: 원지혁 (당근 프론트엔드 코어 팀 리드)
> 출처: https://www.youtube.com/watch?v=4tZbWyeujK8

---

## 발표 개요

당근은 모바일 앱 회사이지만, 핵심 프론트엔드 기술은 **웹뷰**를 만드는 데 집중되어 있다. 당근 프론트엔드 코어 팀은 지난 2년간 웹뷰 개발 도구를 만들면서 얻은 경험을 토대로, 웹뷰 플랫폼을 구성하는 세 가지 큰 개념을 정의했다.

1. **Bridge** — 웹뷰와 네이티브 사이의 통신을 담당하는 모듈
2. **Framework** — 개발자가 일정한 구조 안에서 코드를 더 쉽게 작성하도록 돕는 틀
3. **Platform** — 앱이 실행되는 기반 환경 (배포, 버전 관리 등)

이 글은 각 개념마다 **(1) 어떤 문제를 정의했는지**, **(2) 그 문제를 어떻게 해결했는지** 그 사고 흐름을 정리한다.

---

## 1. Bridge: 협업 비용을 시스템으로 흡수하기

### 문제 정의

처음 브릿지를 만들었을 때(약 3년 전), 새로운 브릿지 함수를 추가하는 절차는 다음과 같았다.

1. 네이티브 개발자와 웹뷰 개발자가 회의를 잡아 인터페이스를 합의
2. 웹뷰 개발자가 합의된 명세로 타입스크립트 SDK를 작성
3. SDK를 사용하는 테스트용 웹뷰를 만들어 배포
4. 네이티브 개발자가 그 URL을 받아 테스트하면서 구현

**관찰된 통증(pain points)**

- **높은 소통 비용**: 회의 자체도 비용이지만, 구현 단계에서 발견되는 인터페이스 결함 때문에 자주 회의 단계로 회귀
- **시간이 지날수록 느려지는 작업**: 단순한 변경에도 풀 사이클이 돌아야 함
- **높은 진입 장벽**: 기여하려면 양쪽 도메인 지식이 필요해서 누구나 쉽게 작업할 수 없음

### 문제 정의의 핵심 패턴

> "이 작업이 본질적으로 사람과 사람의 합의 문제인가, 아니면 **반복 가능한 절차**인가?"

회의/구현/테스트 사이의 왕복은 본질적으로 **명세를 코드로 옮기는 기계적인 작업**이다. 이 부분은 사람이 관여할 필요가 없다.

### 해결 방식

**JSON 스키마를 단일 진실 원천(Single Source of Truth)으로 삼고, 그로부터 SDK와 테스트 웹뷰를 자동 생성**하도록 워크플로우를 재설계.

새 워크플로우:

1. 기능이 필요한 개발자가 브릿지 레포에서 **JSON 스키마 한 파일을 수정해 PR**
2. PR 생성 시 변경 사항이 반영된 **SDK + 테스트용 웹뷰가 자동 배포**
3. 프론트엔드/네이티브 개발자가 스키마 변경을 리뷰
4. 머지되면 정식 SDK와 테스트 웹뷰가 배포되어 모두가 사용 가능

### 배울 점

- **회의가 반복된다는 신호는, 회의에서 나누던 정보가 코드로 표현될 수 있다는 신호다.**
- 인터페이스 정의를 데이터(스키마)로 분리하면, 양쪽이 동일한 산출물을 자동으로 받아볼 수 있다.
- 도구화가 끝나면 "기여 가능한 사람"이 폭발적으로 늘어난다 — 이 자체가 시스템의 가치다.

## 2. Framework (Stackflow): 매체에 맞는 추상화 만들기

### 문제 정의

Stackflow는 처음에는 단순히 **전환 애니메이션과 백 스와이프 같은 모바일 UI를 모사**하기 위한 패키지였다. 하지만 운영하면서 모바일 웹뷰 환경에 맞지 않는 두 가지 한계를 발견했다.

#### 문제 ① UI 라이프사이클 — 같은 UI라도 매체마다 기대되는 동작이 다르다

예: Alert Dialog가 떠 있을 때 뒤로 가기를 누르면?

- **PC 브라우저**: 뒤로 가기에 특별한 기대가 없음
- **모바일**: 백버튼을 누르면 다이얼로그가 닫히기를 기대

→ 같은 모양의 UI라도 **히스토리 관리 방식이 매체에 따라 달라진다**.
→ 모바일에서는 히스토리 관리가 UX에 결정적인 영향을 미친다.

#### 문제 ② 라우팅 — 웹의 URL 모델이 모바일에 그대로 들어맞지 않는다

URL은 path, path parameter, query parameter, hash 같은 구성 요소로 이루어져 있다.
이 체계는 웹에서는 의미 있고 유효하지만, **모바일 화면을 찾아갈 때는 불필요한 복잡성**을 만든다.

> "모바일스럽고 단순한 개념은 없을까?"

### 해결 방식

#### ① 모바일 히스토리에 맞는 라우팅 프레임워크로 역할 확장

Stackflow를 단순 애니메이션 라이브러리에서 **모바일 화면 라우팅 프레임워크**로 정체성을 재정의.

#### ② Activity 개념 도입

화면을 다음 두 가지로만 단순화:

- **이름** (Activity name)
- **파라미터** (Activity parameter)

URL 기반 라우팅이 필요한 경우는, "이 URL을 어떤 Activity 이름과 파라미터로 연결할지" **확장 개념**으로 매핑한다 — Activity가 1차 시민, URL은 부수적.

### 핵심 철학

> "웹뷰 개발을 **웹 개발이 아닌 모바일 개발**로 느끼게 하자."

이 한 문장이 Stackflow 설계의 모든 결정을 지배한다.

### 배울 점

- **기존 도구를 끌어다 쓰지 말고, 매체의 본질에 맞는 추상화를 새로 만들 것.**
- "표준이니까 따른다"가 아니라 "이 매체에서 본질적으로 무엇이 1차 시민이어야 하는가?"를 먼저 묻는다.
- URL → Activity로의 전환은 단순히 API 변경이 아니라, **개발자의 멘탈 모델을 바꾸는 작업**이다.

## 3. Platform (Warp): 빌리기 → 만들기로 가는 진화

### 배포 플랫폼의 역할 재정의

발표자는 배포를 단순한 파일 업로드가 아니라 두 가지 만남의 공간으로 정의한다.

- 한쪽은 **개발자가 제품을 만드는 공간** (버전 관리, 테스트, QA, 실험)
- 다른 한쪽은 **유저가 제품을 만나는 초기 경험** (속도, 안정성)

### 문제 정의의 진화 — 4세대에 걸친 배포 플랫폼

| 세대 | 도구 | 얻은 것 | 발견한 한계 |
|---|---|---|---|
| 1 | AWS S3 + CloudFront | 단순한 정적 파일 서빙 | 버전 관리, 롤백, 프리뷰 등 생산성 기능을 사내 CI로 직접 구축해야 함 |
| 2 | Vercel | 자동 배포, 프리뷰, 롤백 등 생산성 기능 충분 | 트래픽 기반 과금 → 첫 화면 노출/전체 푸시 시 큰 비용. 예산 관리 어려움 |
| 3 | Cloudflare Pages | 트래픽 비용 안정 + 생산성 기능 대부분 충족 | 간헐적으로 다른 국가 엣지로 라우팅되어 성능 이슈 |
| 4 | **Warp (자체 구축)** | 위 모두를 보완 + 정책 관리 + 플러그인 확장성 | (현재 진행 중) |

### 의사결정 패턴

당근은 오랫동안 **"생태계의 잘 만든 플랫폼을 쓰는 것이 자체 구축보다 낫다"** 는 원칙을 고수했다. 이 원칙 덕분에 다양한 플랫폼을 경험하면서 **배포 플랫폼이 정확히 무엇을 해야 하는지를 학습**할 수 있었다.

> 자체 구축은 "처음부터" 하는 것이 아니라, **충분히 기존 솔루션을 써본 뒤 마지막 수단으로** 한다.

### Warp의 설계 결정

자체 구축 시점에서 Warp는 단순한 "Vercel의 사내 클론"이 되지 않았다. 대신 배포 플랫폼의 역할을 두 방향으로 확장했다.

1. **개발자 영역의 정책 관리**: 사내 프론트엔드 정책을 배포 단계에서 일괄 적용 가능
2. **플러그인 인터페이스**: 어떤 워크플로우든 확장해서 사용 가능한 유연한 구조

### 통합 — Bridge × Framework × Platform

데모로 보인 PoC: Stackflow에 Warp 플러그인을 달면, **Activity가 보여질 때마다 자동으로 페이지뷰 이벤트가 수집**된다. 즉, 세 컴포넌트가 유기적으로 연결될 때 진가가 드러난다.

### 배울 점

- **자체 구축 결정은 "기존 도구의 단점이 명확해진 뒤"가 가장 안전하다.**
- 외부 솔루션을 충분히 써보지 않은 자체 구축은 "다시 발명한 더 나쁜 바퀴"가 되기 쉽다.
- 자체 구축의 진짜 가치는 **자사 도메인에 특화된 확장(정책 관리, 플러그인)** 에서 나온다 — 단순 기능 모방이 아니다.

## 종합: 문제 정의와 해결의 패턴

세 가지 사례를 관통하는 공통 패턴은 다음과 같다.

### 1. "사람의 반복 작업"을 "시스템의 자동화"로 옮긴다 (Bridge)

회의·합의·구현의 왕복이 반복된다면, 그 반복은 **추상화 부재의 신호**다. 합의 대상을 명세 파일로 만들고, 명세에서 결과물을 자동 생성하는 순간 협업 비용이 사라진다.

### 2. "기존 표준"을 그대로 쓰지 말고 "매체의 본질"에 맞춰 재정의한다 (Framework)

웹의 URL은 웹에서 본질적이지만, 모바일 웹뷰에서는 잡음이 된다. 매체가 다르면 1차 시민도 달라야 한다. **익숙한 도구를 의심하는 능력**이 좋은 추상화의 출발점이다.

### 3. "직접 만들기"는 마지막 수단이지만, 마지막에는 반드시 필요하다 (Platform)

기존 솔루션을 충분히 써본 뒤에야 진짜 요구사항이 보인다. 그 시점에 만든 자체 도구는 단순한 클론이 아니라 **자사 도메인에 특화된 확장**까지 포함하게 된다.

### 4. 컴포넌트는 따로 잘 만들되, 마지막엔 유기적으로 연결한다

Bridge, Framework, Platform은 각자 독립된 문제를 해결하지만, 통합되었을 때 (예: Stackflow Activity → Warp 플러그인 → 자동 페이지뷰 수집) **개별 가치의 합 이상의 가치**가 발생한다. 좋은 시스템 설계는 각 부품의 경계를 명확히 하면서도 통합 지점을 미리 설계해 둔다.
