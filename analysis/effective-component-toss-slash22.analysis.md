## 1. 한 줄 요약

복잡한 결제 대시보드처럼 **변경이 일상인 제품**에서, "변경을 예측하지 말고 변경에 대응할 수 있는 컴포넌트"를 어떻게 만들 것인가가 발표가 풀려는 진짜 문제다. (원본 §1, §2)

---

## 2. 문제 정의 패턴 (단계별, 발표자가 실제로 한 행위 중심)

1. **감정적 장면에서 출발** — "이번 스프린트에 변경이 발생했어요"라는, 누구나 겪는 통증을 첫 입구로 삼는다. (원본 §1)
2. **통증의 구조성 입증** — "제품이 변경되지 않아도 되는 조건은 단 하나, 사용자가 잘 사용하고 있을 때." 따라서 변경은 일회성이 아니라 일상이라고 못 박는다. (원본 §1)
3. **프레임 전환** — "변경은 미리 알 수 없다 → 예측하지 말고 대응하자." 풀 수 없는 문제(예측)를 풀 수 있는 문제(대응 가능한 구조 만들기)로 재정의. (원본 §2)
4. **자기 코드를 증거로 제시** — 본인이 3년 전에 짠 `Item`, `popUpJungItem`처럼 의미 없는 네이밍을 직접 공개. 추상론 대신 구체로 신뢰를 만든다. (원본 §2)
5. **통증의 메커니즘 분해** — "변경하려면 어디부터 봐야 할지 모른다 → 변경 비용 = 탐색 비용 + 영향 범위 추적 비용"으로 측정 가능한 형태로 쪼갬. (원본 §2)
6. **임의성을 문제 핵심으로 지목** — "만들다 보니 페이지가 커졌고 적당히 덜어냈다"의 "적당히"를 공격 대상으로 삼는다. 자의적 판단 그 자체를 문제로 본다. (원본 §2)

---

## 3. 해결 패턴 (단계별, 어떤 원칙·도구·사고를 적용했나)

1. **범위 한정** — "변경에 제어 가능한 단위 = 컴포넌트"로 해결 무대를 좁힘. (원본 §3 도입)
2. **변하는 것과 변하지 않는 것의 분리 (헤드리스)** — 데이터 관리 / UI 표현 / 상호작용을 각자 다른 수명으로 보고 분리. 데이터 추상화는 훅(`useCalendar`)으로, 상호작용 추상화도 훅(`useLongPress`)으로. (원본 §3 원칙 1)
3. **데이터 흐름이 같으면 인터페이스도 같게** — UI 모양이 달라 보여도 데이터 흐름이 동일하면 같은 합성으로 만든다. Select와 모달+체크박스 다중선택을 같은 `Dropdown` 가족으로 묶음. (원본 §3 원칙 2)
4. **표준 인터페이스로 수렴** — `FrameworkSelect` 대신 `MultiSelect`처럼 도메인 단어를 걷어낸 인터페이스. 사람은 기존 배경 지식으로 새것을 해석하므로, HTML `<select>` 같은 표준에 가까울수록 이해 비용이 낮아진다. (원본 §3 원칙 3)
5. **레이어로 도메인을 흡수** — 도메인을 모르는 일반 컴포넌트와 도메인을 아는 컴포넌트를 레이어로 구분해, 비즈니스 로직은 위에서, UI 로직은 아래에서 처리. (원본 §3 원칙 3)
6. **인터페이스를 먼저, 구현은 나중** — "이미 만들어져 있다고 가정하고" 사용 코드부터 적어본다. 무엇을 받고 무엇을 내보낼지가 어떻게 구현할지보다 먼저 결정되어야 한다. (원본 §4 액션 1)
7. **분리 이유를 다시 묻기** — 복잡도 때문인지, 재사용 때문인지, 진짜로 분리해야 하는지를 묻는다. 이유가 안 나오면 분리하지 않는다. (원본 §4 액션 2)

## 4. 사용된 메타 도구

- **감정 장면 → 구조적 통증으로 격상** — 누구나 공감하는 한 장면을 "예외가 아니라 일상"임을 보여 일반 문제로 끌어올림.
- **풀 수 없는 문제 → 풀 수 있는 문제로 프레임 전환** — "예측"처럼 본질적으로 불가능한 목표를 "대응"처럼 설계 가능한 목표로 바꿈.
- **자기 코드 비판** — 본인의 과거 실패를 증거로 제시해 추상론이 아닌 구체로 신뢰를 확보.
- **통증의 분해** — 추상적 불편을 "탐색 비용 + 영향 범위 추적 비용"처럼 측정 가능한 항으로 쪼갬.
- **임의성을 공격 대상으로 지목** — "적당히" 같은 자의적 판단을 문제의 핵심으로 본다. 일반화의 적은 임의성.
- **변하지 않는 것과 변하는 것의 분리** — 같은 영역 안에서도 수명이 다른 책임을 다른 추상화로 분리.
- **모양이 아닌 데이터 흐름으로 동일성 판단** — 표면이 다른 두 케이스를 데이터 흐름으로 같은 것으로 묶을 수 있는지 보는 시선.
- **표준에의 수렴 = 인지 비용 감축 도구** — 일반화는 미적 취향이 아니라, 협업/이해 비용을 줄이는 명시적 도구.
- **레이어 분리로 도메인 흡수** — 도메인을 아는 층과 모르는 층을 분리해 각 층이 한 가지 일에만 집중하도록 함.
- **인터페이스 우선 사고 (사용 코드부터 쓰기)** — "이미 있다고 가정하고 써본다"는 사고 실험.
- **"왜 분리하는가"를 다시 묻기** — 행위에 들어가기 전 의도를 한 번 더 확인하는 점검 질문.

---

## 5. 핵심 한 줄

> **변경을 예측하지 말고 대응하라. 그 대응의 단위는 변하지 않는 데이터 흐름을 표준 인터페이스로 굳히고, 변하는 표현·도메인은 별도 층으로 흡수한 컴포넌트다.** (원본 §6)
