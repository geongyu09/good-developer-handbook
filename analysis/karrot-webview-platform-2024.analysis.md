## 1. 한 줄 요약

당근이 2년간 웹뷰 개발 도구를 만들며 정리한 세 축 — Bridge(통신), Framework(틀), Platform(배포 환경) — 각각이 어떤 통증에서 출발해 어떤 추상화로 옮겨갔는지를 보여 주며, 좋은 시스템은 **반복되는 협업을 도구로**, **익숙한 표준을 매체에 맞게 재정의로**, **자체 구축은 마지막 수단으로** 만들어진다는 것을 정리한 발표다. (원본 발표 개요, 종합)

---

## 2. 문제 정의 패턴 (단계별, 발표자가 실제로 한 행위 중심)

### Bridge

1. **현재 절차를 그대로 펼쳐 보이기** — 새 함수 추가 시 회의 → SDK 작성 → 테스트 웹뷰 배포 → 네이티브 구현이라는 4단계 풀 사이클을 가시화. (원본 §1 문제 정의)
2. **통증을 항목으로 분해** — 높은 소통 비용 / 시간이 지날수록 느려지는 작업 / 높은 진입 장벽으로 쪼갬. (원본 §1)
3. **본질 질문 던지기** — "이 작업이 사람과 사람의 합의 문제인가, 아니면 반복 가능한 절차인가?" 회의의 본질이 명세를 코드로 옮기는 기계적 작업임을 폭로. (원본 §1 핵심 패턴)

### Framework (Stackflow)

1. **같은 UI라도 매체마다 기대가 다름을 짚음** — Alert Dialog의 뒤로 가기는 PC와 모바일에서 기대 동작이 다르다. (원본 §2 문제 ①)
2. **기존 표준의 부적합 진단** — 웹의 URL(path/query/hash)은 웹에서는 의미 있지만 모바일 화면에는 불필요한 복잡성. (원본 §2 문제 ②)
3. **본질을 묻는 질문으로 좁히기** — "모바일스럽고 단순한 개념은 없을까?" "이 매체에서 본질적으로 무엇이 1차 시민이어야 하는가?" (원본 §2)

### Platform (Warp)

1. **세대별로 도구를 직접 써본 결과를 펼침** — S3+CloudFront → Vercel → Cloudflare Pages 각각의 얻은 것과 발견한 한계를 표로 정리. (원본 §3 표)
2. **배포의 역할 재정의** — 배포는 파일 업로드가 아니라 "개발자가 제품을 만드는 공간"과 "유저가 제품을 만나는 초기 경험"의 만남이라고 다시 정의. (원본 §3 배포 플랫폼의 역할 재정의)
3. **자체 구축의 시점 기준** — "기존 솔루션을 충분히 써본 뒤 마지막 수단으로." 충분히 써보지 않은 자체 구축은 더 나쁜 바퀴를 다시 발명한다는 위험을 명시. (원본 §3 의사결정 패턴, 배울 점)

---

## 3. 해결 패턴 (단계별, 어떤 원칙·도구·사고를 적용했나)

### Bridge

1. **단일 진실 원천(SSOT)을 데이터로 분리** — 합의 대상을 JSON 스키마 한 파일로 모아 단일 원천으로 삼음. (원본 §1 해결)
2. **명세에서 산출물을 자동 생성** — 스키마 PR이 올라오면 SDK + 테스트 웹뷰가 자동 배포. 사람의 반복 작업을 시스템이 흡수. (원본 §1 새 워크플로우)
3. **기여 가능한 사람의 폭발적 확장** — 도구화의 부산물로 "양쪽 도메인 지식이 있어야 했던" 진입 장벽이 낮아짐. (원본 §1 배울 점)

### Framework (Stackflow)

1. **정체성 재정의** — 단순 애니메이션 라이브러리에서 "모바일 화면 라우팅 프레임워크"로 도구 자체의 정의를 바꿈. (원본 §2 해결 ①)
2. **새 1차 시민 도입 (Activity)** — 화면을 "이름 + 파라미터"라는 두 요소로만 단순화. URL은 부수적 매핑으로 강등. (원본 §2 해결 ②)
3. **하나의 철학으로 모든 결정을 정렬** — "웹뷰 개발을 웹 개발이 아닌 모바일 개발로 느끼게 하자"가 모든 설계 결정을 지배. (원본 §2 핵심 철학)

### Platform (Warp)

1. **외부 솔루션을 충분히 학습 도구로 활용** — 4세대에 걸친 외부 도구 사용으로 "배포 플랫폼이 정확히 무엇을 해야 하는가"를 학습. (원본 §3 의사결정 패턴)
2. **자체 구축은 마지막 수단** — 마지막 수단이지만, 그 시점에는 단순 클론이 아니라 자사 도메인 특화 확장(정책 관리, 플러그인)을 포함. (원본 §3 Warp의 설계 결정, 배울 점)
3. **컴포넌트 간 통합 지점 미리 설계** — Bridge × Framework × Platform이 따로 잘 만들어지되, Stackflow Activity → Warp 플러그인 → 자동 페이지뷰 수집처럼 통합되었을 때 합 이상의 가치를 내도록 설계. (원본 §3 통합, 종합 4)

## 4. 사용된 메타 도구

- **현재 절차의 가시화** — 통증을 말하기 전에 지금 절차를 단계별로 펼쳐 보이는 것 자체가 문제 정의의 1단계.
- **본질 질문으로 통증 분류** — "사람의 합의 문제인가, 반복 가능한 절차인가?"처럼 한 질문으로 통증을 두 종류로 가른다.
- **반복은 추상화 부재의 신호로 읽기** — 회의가 반복된다 = 회의에서 다루던 정보가 코드로 표현될 수 있다는 신호.
- **단일 진실 원천(SSOT)으로 합의를 데이터화** — 합의 대상을 코드/문서가 아닌 한 파일의 데이터로 모음.
- **명세에서 산출물 자동 생성** — 사람이 옮기던 것을 도구가 옮기게 한다. 도구화의 부산물은 기여자 풀의 확장.
- **매체에 맞는 1차 시민 재선정** — 익숙한 표준이 다른 매체에서 잡음이 되는지 묻고, 그 매체의 본질에 맞는 새 1차 시민을 도입.
- **하나의 철학을 정렬축으로 사용** — 한 문장의 철학을 만들고, 모든 설계 결정이 그 문장을 향하도록 정렬.
- **외부 도구를 학습 도구로 활용** — 외부 솔루션을 단순히 쓰는 게 아니라 "내 진짜 요구사항이 무엇인지" 발견하는 도구로 사용.
- **자체 구축은 마지막 수단 (충분히 써본 뒤)** — 충분히 써보지 않은 자체 구축의 위험("더 나쁜 바퀴")을 명시적으로 인지.
- **자체 구축의 가치는 도메인 특화 확장에서** — 단순 기능 모방이 아니라 자사 정책·플러그인처럼 외부가 못 주는 부분에서 가치를 만든다.
- **부품의 경계는 명확히, 통합 지점은 미리 설계** — 각 컴포넌트가 독립적으로 잘 동작하면서도 통합 시 합 이상의 가치가 나오도록 미리 설계.
- **익숙한 도구를 의심하는 능력** — 좋은 추상화의 출발점은 표준을 따르는 게 아니라 "이 표준이 여기서도 맞는가"를 의심하는 것.

---

## 5. 핵심 한 줄

> **반복되는 협업은 단일 진실 원천에서 자동 생성되는 도구로 흡수하고, 익숙한 표준은 매체의 본질에 맞춰 1차 시민을 재선정해 추상화하며, 자체 구축은 외부를 충분히 써본 뒤 자사 도메인 특화 확장으로만 정당화한다.** (원본 한 줄 요약)
