## 1. 한 줄 요약

"빠른 실험이 어렵다"는 본질적 한계 때문에 동네생활을 네이티브 → 웹뷰로 **역방향 전환**한 팀이, 네이티브와 비교 가능한 속도를 만들기 위해 CSR → SSR → 스트리밍 SSR로 의사결정을 좁혀가고, 그 결정이 만든 새 운영 비용까지 정직하게 흡수해 가는 과정을 정리한 발표다. (원본 발표 개요)

---

## 2. 문제 정의 패턴 (단계별, 발표자가 실제로 한 행위 중심)

1. **본질적 한계의 명시** — "잘 동작하지만 빠른 실험과 잦은 UI 변경이 어렵다"는, 기능이 아니라 사이클의 문제로 시작점을 잡음. (원본 §1)
2. **위험이 적은 곳에서 효과 입증** — 변화가 잦은 글쓰기/상세부터 작게 전환해 AB 테스트 데이터로 효과를 증명하고, 그 결과로 진입점(피드)까지 확장 결정. (원본 §1 의사결정 흐름)
3. **비교 기준을 명시** — "비교 대상이 네이티브이기 때문에 최대한 빠르게." 한 문장이 모든 기술 선택의 기준이 됨. 추상적인 "느리지 않게"가 아니라 비교 대상을 명확히 함. (원본 §2)
4. **추상적 불만을 메커니즘으로 분해** — "느리다"를 (a) CDN 리전 라우팅으로 인한 White Screen, (b) Cross-domain Preflight 100ms × N개 누적, (c) 모바일 네트워크 변동성 세 항으로 쪼갬. (원본 §2)
5. **해결책이 만든 새 문제를 다시 정의** — SSR 도입 후 "API가 느리면 CSR보다 더 늦게 보인다"는 새 통증을 명시하고 다음 라운드의 문제로 끌어올림. (원본 §4 새로운 문제 정의)
6. **"당연히 그럴 것" 가정 검증** — "웹뷰니까 배포만 하면 자동 전파"라는 나이브한 가정을 iOS 백그라운드 유지 패턴으로 깨뜨리고 가정을 다시 점검. (원본 §5-4)
7. **해결책이 깨지는 시나리오를 한 단계 더** — 새로고침 유도 해결책이 만든 다음 엣지(오프라인에서 reload 시 웹뷰 먹통)까지 한 번 더 파고듦. (원본 §5-5)

## 3. 해결 패턴 (단계별, 어떤 원칙·도구·사고를 적용했나)

1. **점진적 마이그레이션** — 큰 결정을 한 번에 내리지 않고, 작은 영역에서 데이터로 효과를 증명한 뒤 핵심으로 확장. (원본 §1)
2. **메커니즘을 직접 지우는 도구 선택** — "트렌드"나 "SEO" 같은 일반론이 아니라, "CSR의 두 약점을 직접 지우는가"를 기준으로 SSR을 선택. Next/Remix 같은 풀 프레임워크가 아닌 Vite + React Server로 직접 구축. (원본 §3)
3. **동일 생태계의 기본기로 재해결** — SSR이 만든 올오어낫싱 렌더링 문제를, 새 패러다임이 아니라 React `<Suspense>` + `renderToPipeableStream`이라는 기본 도구로 다시 해결. (원본 §4)
4. **Shell + 비동기 영역 분리** — 화면을 "API와 무관한 부분(Shell)"과 "API가 필요한 부분"으로 나눠 점진적으로 흘려보냄. 한 번에 그리지 않는다. (원본 §4)
5. **운영 비용을 솔직히 명시** — SSR 도입은 노드 서버 운영 부담의 도입이라는 사실을 숨기지 않고 "다른 팀에 권하기 어렵다"고 인정. (원본 §5-2)
6. **로그·메트릭 동일성까지 마이그레이션의 끝점에 포함** — 안드/iOS 로깅 기준 차이로 데이터를 맞추는 시간이 개발 시간보다 길어짐. 기능 동일성만으로는 비교 가능한 출시가 안 된다. (원본 §5-3)
7. **강제 대신 사용자 친화적 권유** — 버전 불일치 시 강제 새로고침이 아니라 토스트로 권유. `visibilitychange`로 백그라운드 복귀 시점에만 비교. (원본 §5-4)
8. **온라인/오프라인 신호로 다음 엣지 처리** — `window.online/offline` 커스텀 훅으로 reload 시점의 네트워크 상태를 분기. 오프라인 reload 시도는 별도 로깅 큐로 보존. (원본 §5-5)
9. **흑마술적 우회를 인정하고 리스크를 명시** — iOS Safari 키보드처럼 정공법이 안 되는 영역은 우회로 풀고 "언제 깨질지 모른다"는 리스크를 명시적으로 관리. (원본 §5-1)

## 4. 사용된 메타 도구

- **사이클 관점의 문제 정의** — "기능이 동작하는가"가 아니라 "실험·검증 사이클이 도는가"를 통증의 단위로 본다.
- **점진적 마이그레이션 (저위험 영역에서 데이터로 증명)** — 위험이 적은 영역을 먼저 옮기고, 그 결과를 핵심 영역 확장 결정의 근거로 쓴다.
- **비교 기준의 명시** — "빠르게"가 아니라 "무엇과 비교했을 때"를 못 박아 모든 후속 결정의 기준선을 만든다.
- **추상적 통증의 메커니즘 분해** — "느리다"를 단계 × 메커니즘 × 숫자(100ms)로 환원해 측정 가능하게 만든다.
- **메커니즘 매핑 기반 도구 선택** — 트렌드나 일반론이 아니라 "이 도구가 우리 문제의 메커니즘을 직접 지우는가"로 채택을 결정.
- **해결책이 만든 새 문제 → 동일 생태계 기본기로 재해결** — 새 패러다임을 도입하지 않고, 같은 도구의 다음 기본기로 한 단계 더 깊이 풀어낸다.
- **Shell / 비동기 영역의 분리** — 응답 가능한 부분과 기다려야 하는 부분을 다른 수명으로 보고 따로 흘려보냄.
- **운영 비용의 명시화** — 기술 도입 = 운영 부담 도입. 기술 평가에 운영 부담을 같은 항목으로 포함.
- **마이그레이션의 끝점 = 메트릭 동일성** — 기능 동일성만으로는 끝나지 않는다. 로그·지표가 같아야 비교 가능한 출시가 된다.
- **"당연히 그럴 것" 가정의 검증** — 익숙한 가정을 실제 사용자 행동·플랫폼 동작으로 한 번 더 검증.
- **해결책이 깨지는 시나리오 한 단계 더 묻기** — 어떤 해결책이든 다음 엣지를 한 번 더 묻는 습관.
- **강제보다 유도** — 사용자에게 강제하지 않고 신호(가시성 변화, 네트워크 상태)에 맞춰 권유.
- **흑마술 인정 + 리스크 명시** — 정공법이 없는 영역에서는 우회를 인정하고, 깨질 가능성을 관리 대상으로 등록.

---

## 5. 핵심 한 줄

> **"느리다"를 메커니즘 × 숫자로 분해하고, 그 메커니즘을 직접 지우는 도구를 골라야 한다. 그리고 그 선택이 만든 새 문제·운영 비용·다음 엣지까지 한 단계씩 더 묻는 사람만이 끝까지 책임진 마이그레이션을 한다.** (원본 부록 패턴 표 종합)
