# 지혜로운 해결 패턴 카탈로그

`10-pattern-convergence.md`의 B·D 섹션과 `analysis/*.analysis.md`의 "3. 해결 패턴", "4. 사용된 메타 도구"를 입력으로, 통증을 풀 때 꺼내 쓸 수 있는 8개의 재사용 패턴.

## "지혜롭다"의 조작적 정의

각 패턴이 "왜 지혜로운가"를 판단하는 기준이 되는 4가지 정의:

1. **더 작은 결정으로 더 많은 변화를 흡수**
2. **자의적 판단을 측정 가능한 기준으로 바꿈**
3. **단기 통증을 감수하더라도 시스템의 본질을 단순화함**
4. **협업 비용·인지 비용을 코드 구조로 흡수함**

각 패턴은 위 정의 중 하나 이상을 만족해야 한다.

---

## 패턴 1. 수명별 분리 (Lifetime-Based Separation)

- **핵심 메커니즘**: 정의 ① + ④. 한 단위 안의 책임 중 변하는 속도가 다른 것을 분리해, 짧은 수명의 변경이 긴 수명을 흔들지 않게 한다. 한 번의 분리 결정으로 양쪽이 독립적으로 진화 — 작은 결정으로 큰 변화 흡수.
- **적용 절차**:
  1. 한 단위 안에 묶인 책임들의 **변경 빈도(수명)** 를 한 줄씩 평가한다.
  2. 수명이 가장 다른 두 책임 쌍을 식별한다.
  3. 수명이 짧은 쪽을 별도 단위로 분리한다 (훅·서브컴포넌트·서비스 레이어 등 — 단위는 매체에 맞게).
  4. 분리 후 두 단위가 서로의 사정을 모르도록 인터페이스를 좁힌다.
- **출처 사례**: 헤드리스 패턴으로 데이터(`useCalendar`)·UI·상호작용(`useLongPress`)을 각자 다른 수명으로 분리 (E §3 원칙 1). API 응답과 무관한 Shell을 먼저, API 필요한 부분은 `<Suspense>`로 나중에 흘려보내는 스트리밍 SSR (K1 §4). 합의는 JSON 스키마(데이터)에, SDK는 자동 생성(산출물)에 (K2 §1).
- **흔한 오용**: "분리 자체가 목적"이 되어 모든 것을 잘게 나눈 결과 통신 비용이 분리 이익을 초과. 수명이 같은 책임까지 분리해 무의미한 추상 레이어를 만듦.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 1 (Visibility-First — 분리 후보를 펼쳐 봐야 수명이 보인다).

## 패턴 2. 인터페이스 우선 결정 (Interface-Before-Implementation)

- **핵심 메커니즘**: 정의 ② + ④. 구현 사정이 사용자 면을 오염시키는 것을 막아, 호출부의 자의적 의존을 줄임. 인터페이스 결정에 시간을 먼저 들임으로써 협업·이해 비용을 코드 구조로 흡수.
- **적용 절차**:
  1. 만들 단위가 **이미 만들어져 있다고 가정**한다.
  2. 사용 코드부터 적어 본다 — 호출부, 타입, 시그니처.
  3. 사용 코드가 어색하면 인터페이스를 다듬는다 — 구현은 아직 시작하지 않는다.
  4. 인터페이스가 굳어진 뒤에야 구현으로 들어간다.
- **출처 사례**: "이미 만들어져 있다고 가정하고 사용 코드부터 작성. 무엇을 받고 무엇을 내보낼지가 어떻게 구현할지보다 먼저 결정되어야 한다" (E §4). JSON 스키마 한 파일 PR이 먼저, SDK + 테스트 웹뷰는 자동 생성 (K2 §1).
- **흔한 오용**: 진짜 요구사항을 모르는 영역에서 인터페이스를 너무 일찍 굳혀 잘못된 추상에 갇힘. 학습이 부족할수록 인터페이스 결정도 잠정적이어야 한다.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 4 (Essential Question Framing — 본질 질문이 인터페이스 결정의 입구).

## 패턴 3. 메커니즘 매핑 기반 도구 채택 (Mechanism-Mapped Adoption)

- **핵심 메커니즘**: 정의 ②. 도구 채택을 "트렌드"·"의리"라는 자의적 판단에서 "이 도구가 우리의 X 메커니즘을 직접 지우는가"라는 측정 가능한 기준으로 바꿈.
- **적용 절차**:
  1. 우리 통증을 메커니즘으로 분해해 둔다 (문제 정의 패턴 2의 결과물).
  2. 후보 도구 각각이 **어느 메커니즘을 직접 지우는지** 한 줄씩 적는다.
  3. 가장 큰 메커니즘을 직접 지우는 도구만 후보에 남긴다.
  4. 채택 시 매핑을 1줄 코멘트로 코드/문서에 박아 둔다 — 1년 뒤에도 검증 가능하도록.
- **출처 사례**: SSR을 채택한 근거가 "CSR의 CDN 리전 + Preflight × N개 + 네트워크 변동성을 직접 지움"인 매핑 (K1 §3). Vercel·Cloudflare·자체 구축의 4세대 트레이드오프를 표로 비교해 채택 시점을 결정 (K2 §3).
- **흔한 오용**: 매핑을 거창하게 만들어 도구 채택 자체가 무거워짐. 매핑은 1줄이면 충분 — 매핑의 무게가 채택의 무게를 초과해선 안 된다.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 2 (Mechanism Decomposition — 분해된 통증이 매핑의 직접 입력).

## 패턴 4. 한 문장 정렬축 (Single-Sentence Alignment Axis)

- **핵심 메커니즘**: 정의 ② + ④. 후속 수십 개 결정의 모호함을 한 문장이 지운다. 자의적 판단의 누적을 한 문장으로 흡수해 인지 비용을 코드 구조 위쪽으로 끌어올림.
- **적용 절차**:
  1. 결정의 모호함이 큰 입구를 식별한다 (대규모 결정의 시작점).
  2. 한 문장의 정렬축을 만든다 — **비교 대상**이나 **매체 명시** 같은 구체성을 반드시 포함.
  3. 후속 결정마다 그 문장을 다시 꺼내 본다.
  4. 정렬축이 흐려지면 한 번 더 다듬는다 — 정렬축은 살아 있는 문서.
- **출처 사례**: "비교 대상이 네이티브이기 때문에 최대한 빠르게" — 이 한 문장이 모든 기술 선택의 기준 (K1 §2). "웹뷰 개발을 웹 개발이 아닌 모바일 개발로 느끼게" — Stackflow 설계의 모든 결정이 이 문장을 향함 (K2 §2).
- **흔한 오용**: 정렬축이 추상적("좋은 코드를 짜자")이면 정렬 효과가 0. 정렬축을 만들어 두고 문서에만 박아 후속 결정에서 안 꺼내는 상태도 흔함.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 4 (Essential Question Framing — 본질 질문이 정렬축의 씨앗).

## 패턴 5. 부품 독립 + 통합 지점 사전 설계 (Decoupled Composition with Pre-Designed Integration)

- **핵심 메커니즘**: 정의 ①. 각 부품이 서로의 존재를 모르고도 동작하면 변화를 부품 단위로 흡수할 수 있다. 통합 지점이 사전에 설계되어 있으면 합 이상의 가치가 자연 발생 — 한 번의 설계 결정이 양쪽 부품의 진화를 흡수.
- **적용 절차**:
  1. 합쳐질 후보 부품들을 식별한다 (둘 이상에서 같이 쓰일 부품).
  2. 각 부품의 책임을 좁혀, 다른 부품의 존재를 모르고도 동작하도록 인터페이스를 만든다.
  3. 부품들이 만나는 지점을 미리 설계 — 데이터 흐름 / 이벤트 / 플러그인 등의 형태로.
  4. 통합 지점이 양쪽 부품의 인터페이스를 오염시키지 않도록 한다.
- **출처 사례**: `Dropdown` · `DropdownTrigger` · `DropdownMenu` · `DropdownItem`이 서로의 존재를 모르되 합성 가능 (E §3 원칙 2). Bridge × Framework × Platform이 따로 잘 만들어지되 Stackflow Activity → Warp 플러그인으로 자동 페이지뷰 수집되는 통합 (K2 §3).
- **흔한 오용**: 부품을 잘게 만들어 놓고 통합 지점이 사후에 끼워 맞춰져 양쪽 인터페이스가 오염됨. 또는 일회성 코드에 통합 설계까지 들여 비용을 낭비.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 3 (Assumption Audit — 익숙한 통합 방식이 매체에 맞는지 의심).

## 패턴 6. 단일 진실 원천 + 자동 생성 (Single Source of Truth with Auto-generation)

- **핵심 메커니즘**: 정의 ④ + ①. 회의·합의·수동 절차로 흩어져 있던 정보를 한 데이터 원천으로 모으고, 그 원천에서 산출물을 자동 생성. 협업 비용을 코드 구조(스키마 + 파이프라인)로 흡수, 한 번의 도구화로 무한 반복되던 비용을 0에 가깝게.
- **적용 절차**:
  1. 반복되는 협업·합의의 대상을 식별한다 — 반복 횟수가 임계 이상인지 확인.
  2. 그 대상을 데이터(스키마·설정 파일 등) 한 곳으로 모은다 (SSOT).
  3. SSOT에서 필요한 산출물(SDK, 테스트, 문서 등)을 자동 생성하는 파이프라인을 만든다.
  4. 사람의 합의를 SSOT 변경 PR로 옮긴다 — 회의가 아닌 코드 리뷰로.
- **출처 사례**: 새 함수 추가 시 JSON 스키마 한 파일을 PR하면 SDK + 테스트 웹뷰가 자동 배포 (K2 §1). 결과: "양쪽 도메인 지식이 필요해 진입 장벽이 높던" 작업이 누구나 기여 가능해짐.
- **흔한 오용**: 빈도가 낮거나 매번 미묘하게 다른 영역에서 SSOT + 자동 생성을 만들어 도구화 비용이 가치를 초과. 또는 SSOT가 충분히 표현력이 없어 결국 사람의 보강이 더 필요해진 상태.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 5 (Reframing — "사람의 합의 → 데이터화"의 프레임 전환이 이 패턴의 입구).

## 패턴 7. 외부를 학습 도구로, 자체 구축은 마지막 수단 (External-as-Learning-Tool)

- **핵심 메커니즘**: 정의 ③. 자체 구축의 매혹을 단기 비용으로 감수하고, 외부 솔루션을 학습 도구로 충분히 운영해 진짜 요구사항이 보일 때까지 기다림. 결과적으로 자체 구축의 본질이 단순해진다 — 클론이 아닌 도메인 특화 확장만 만든다.
- **적용 절차**:
  1. 새 영역의 도구가 필요할 때, **자체 구축의 유혹부터 의식적으로 보류**.
  2. 외부 솔루션을 충분히 운영하며 진짜 요구사항을 학습 (1세대~N세대).
  3. 외부 솔루션이 풀지 못하는 부분이 명확해지는 시점을 기다린다.
  4. 자체 구축은 그 부분에 집중 — 외부 도구의 클론이 아닌 자사 도메인 특화 확장에서 가치를 만든다.
- **출처 사례**: AWS S3 + CloudFront → Vercel → Cloudflare Pages → 자체(Warp)의 4세대 진화. Warp의 가치는 단순 Vercel 클론이 아니라 사내 정책 관리 + 플러그인 인터페이스라는 도메인 특화 확장에서 나옴 (K2 §3).
- **흔한 오용**: "충분히 써본"의 정의가 모호해 사실상 한 번 써보고 자체 구축으로 넘어감. 또는 자체 구축에 들어가서도 외부 솔루션의 클론을 만들어 가치 추가가 0.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 6 (Next-Round Inclusion — 외부 학습이 다음 라운드 사고의 일부).

## 패턴 8. 다음 라운드까지 같은 사고에 포함 (Next-Round-In-Same-Thought)

- **핵심 메커니즘**: 정의 ③ + ④. 단기 통증(운영 부담 인정, 한계 솔직, 메트릭 동일성 비용)을 같은 사고에 포함하면서도 시스템의 본질을 끝까지 책임진다. 사후 비용을 미리 코드/구조로 흡수해 사용자·운영자에게 떠넘기지 않는다.
- **적용 절차**:
  1. 해결책 결정 직후 **새 비용·엣지·운영 부담**을 한 줄씩 적는다.
  2. **메트릭 동일성**이 새 환경에서도 확보 가능한지 확인 — 비교 가능한 출시의 끝점.
  3. 해결책이 깨지는 시나리오 1개를 미리 처리하거나 명시적으로 보류 표시.
  4. **강제보다는 신호 기반 유도**로 사용자 친화성을 확보 (가시성 변화, 네트워크 상태 등의 신호 활용).
  5. 정공법이 안 되는 영역의 **흑마술적 우회**는 인정하고 리스크를 관리 대상에 등록.
- **출처 사례**: SSR 도입 → 스트리밍 SSR → 운영 부담 명시 → 메트릭 동일성 → `visibilitychange` 기반 새로고침 권유 → 오프라인 reload 처리까지 같은 발표에 모두 포함 (K1 §4 ~ §5). iOS Safari 키보드 처리는 흑마술임을 인정하고 리스크 명시 (K1 §5-1).
- **흔한 오용**: 다음 라운드를 이번 라운드에 다 풀려 해서 결정이 무한 미뤄짐. 또는 다음 라운드를 적어 두고 후속 추적이 없어 메모로만 남음 — 보류 항목에는 명시적 owner와 trigger 조건이 있어야 한다.
- **함께 쓰면 좋은 패턴**: 문제 정의 패턴 6 (Next-Round Inclusion — 문제 정의 단계와 해결 단계 양쪽에서 같은 사고가 작동).

## 안티패턴 짧은 목록

세 발표에서 발표자가 직접 "이렇게 하면 안 된다"고 짚은 행위를 일반화한 것:

- **"적당히" 분리** — 이유가 안 나오는데도 "크다"는 이유로 단위를 쪼개는 습관. 결과는 `Item`·`popUpJungItem` 같은 의미 없는 단위가 늘어나는 것. (E §2, §4 액션 2)
- **트렌드 채택** — 도구의 메커니즘 매핑 없이 "이게 좋대"로 도입. SSR 채택 근거가 "트렌드"나 "SEO"가 아니라 매핑이어야 한다는 K1의 명시적 거부. (K1 §3)
- **회의로 푸는 합의 반복** — 반복되는 합의를 도구화하지 않고 사람으로 푸는 시도. K2가 "회의가 반복된다 = 추상화 부재의 신호"라고 직접 진단. (K2 §1)
- **강제 갱신** — 사용자 친화 신호 없이 강제 새로고침·강제 마이그. K1은 강제 대신 토스트로 권유. (K1 §5-4)
- **흑마술 묵인** — 정공법이 안 되는 우회를 묵인하고 리스크 등록 안 함. K1은 흑마술임을 솔직히 인정하고 리스크를 명시. (K1 §5-1)
- **클론 자체 구축** — 외부 솔루션의 단순 모방을 자체 구축이라 부름. K2는 자체 구축의 가치는 도메인 특화 확장에서만 나온다고 못 박음. (K2 §3 배울 점)
- **메트릭 동일성 무시** — 기능 동일성만 맞추고 마이그 종료를 선언. K1은 안드/iOS 로깅 차이로 "개발 시간보다 데이터를 맞추는 시간이 더 오래 걸렸다"고 직접 경고. (K1 §5-3)
- **운영 비용 은폐** — 좋은 기술의 도입을 권하면서 운영 부담은 안 드러냄. K1이 "다른 팀에 무작정 권하기 어려운 이유"로 명시적으로 운영 부담을 같은 평가 항목에 포함. (K1 §5-2)
