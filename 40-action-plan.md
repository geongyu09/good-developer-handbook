# 우테코 크루·개발자 지망생을 위한 액션 플랜

`20-abstract-rules.md`, `30-problem-definition-patterns.md`, `31-wise-solution-patterns.md` 세 산출물을 일상 행위로 번역한 20개 액션. 각 액션은 본인 혼자 시작 가능하고, 5~30분 안에 첫 산출물이 나오게 설계되어 있다.

---

## 0. 이 액션 플랜의 사용법

- **한 번에 다 적용하지 말고, 한 사이클(1주)에 1~2개만 새로 시도**한다. 룰 1·9·6의 우선순위(20-abstract-rules.md 참고)를 따라 들어가는 것이 가장 빠르다.
- 모든 액션에는 **종이/노트/PR 본문 같은 외부 산출물**이 있어야 한다. 머릿속에서만 한 액션은 안 한 것과 같다.
- 한 주를 끝낼 때 **W1만 빠뜨리지 않는다**. W1은 다른 액션의 누적을 점검하는 메타 액션이다.
- 안티 액션(§6)은 시작 시점이 아니라 **결정 직전의 STOP 사인**으로 쓴다. "지금 이걸 하려는데 안티 액션 중 어디 해당하지?"를 묻는 형식.

---

## 1. 매일 단위 (Daily)

### 액션 D1. "느리다·복잡하다·어렵다" 통증을 만나면 즉시 메커니즘으로 분해해 종이에 적기

- **연결되는 룰/패턴**: Rule 1, 문제 정의 패턴 2 (Mechanism Decomposition).
- **방법**: 통증을 한 단어로 말하고 싶어지는 순간 멈추고, **"단계 × 메커니즘 × 측정"** 3열 표를 종이/노트에 그린다. 항을 최소 2개 이상 채울 때까지 손을 놓지 않는다. 숫자가 안 잡히면 추측 숫자라도 적는다 (예: "체감 5초").
- **확인 방법**: 그날의 노트에 통증 분해 표 1개 이상 — 한 주 모이면 5~7장이 노트에 쌓인다.
- **첫 1주 권장 횟수**: **매일 1회 이상**. 통증을 못 찾았다면 PR/페어 중 자의적 판단을 한 곳을 통증으로 쳐도 된다.

### 액션 D2. 새 함수·컴포넌트를 만들기 전, "사용 코드 3줄"을 IDE에 먼저 적기

- **연결되는 룰/패턴**: Rule 6, 해결 패턴 2 (Interface-Before-Implementation).
- **방법**: 구현에 들어가기 전, 그 단위를 **호출하는 코드 3줄**을 빈 파일이나 주석으로 먼저 쓴다. 호출이 어색하면 시그니처/타입을 다듬고, 다듬는 동안 구현은 시작하지 않는다.
- **확인 방법**: 커밋 히스토리에 "시그니처 결정 → 구현" 순서가 보임. 또는 PR 본문에 "이 단위는 이렇게 부르고 싶었다 → 그래서 X 시그니처"의 한 줄.
- **첫 1주 권장 횟수**: 새 함수/컴포넌트마다 = **매일 2~3회**.

### 액션 D3. 단위를 분리하려는 순간 "왜 분리하는가"를 한 줄로 적기

- **연결되는 룰/패턴**: Rule 5 (한계), 안티 액션 A1 ("적당히" 분리 회피).
- **방법**: 함수/컴포넌트/모듈을 새로 분리하려는 순간 손가락을 멈추고, **"왜 분리하는가"** 한 줄을 답한다. 1줄로 답이 안 나오면 분리하지 않고 합쳐 둔다.
- **확인 방법**: 분리한 단위마다 PR 본문이나 코드 주석에 분리 이유 1줄. 합쳐 둔 결정도 회고 노트에 1줄.
- **첫 1주 권장 횟수**: **매 분리 시점** = 매일 1~2회.

## 2. 매 PR / 매 과제 단위

### 액션 P1. PR 본문 첫 줄에 "통증의 메커니즘"을 1줄로 적기

- **연결되는 룰/패턴**: Rule 1, Rule 7, 해결 패턴 3 (Mechanism-Mapped Adoption).
- **방법**: PR 제목 아래 첫 줄에 다음 형식으로 1줄을 적는다.
  > **통증**: X가 (어느 단계에서 / 어떤 메커니즘으로 / 얼마나) → **이 PR**이 그 메커니즘의 N번째 항을 직접 지움.
- **확인 방법**: 모든 PR이 첫 줄에 이 매핑을 가짐. GitHub PR 검색으로 "통증" 키워드가 있는 PR 비율을 직접 셀 수 있음.
- **첫 1주 권장 횟수**: **매 PR**.

### 액션 P2. PR 머지 직전 "이 변경이 만든 다음 라운드 통증 1개"를 본문 끝에 적기

- **연결되는 룰/패턴**: Rule 9, 문제 정의 패턴 6 (Next-Round Inclusion), 해결 패턴 8.
- **방법**: 머지 버튼 누르기 전, 본문 마지막 단락에 "이 변경이 만들 새 비용/엣지/운영 부담 **1개**"를 적는다. 이번 PR에서 안 풀더라도 적는 것이 목적.
- **확인 방법**: 모든 PR 본문 마지막에 "다음 라운드" 단락. 단락이 비어 있으면 머지 안 함.
- **첫 1주 권장 횟수**: **매 PR**.

### 액션 P3. PR 자기 코멘트에 "이번에 적당히 한 부분 1개"를 직접 짚기

- **연결되는 룰/패턴**: Rule 1 (가시화), 안티 액션 A1.
- **방법**: PR을 올린 직후, 자기 PR에 자기 코멘트를 단다. "이 PR에서 내가 자의적으로 판단한 부분 1개"를 직접 짚는다 — 리뷰어가 찾을 때까지 기다리지 않는다.
- **확인 방법**: 모든 PR에 본인의 자기 비판 코멘트 1개.
- **첫 1주 권장 횟수**: **매 PR**.

### 액션 P4. 새 의존성·도구를 도입하는 PR에는 "메커니즘 매핑 1줄"을 코드 주석으로 박기

- **연결되는 룰/패턴**: Rule 7, 해결 패턴 3.
- **방법**: 새 라이브러리/도구를 추가할 때, "이 도구가 우리 통증의 X 메커니즘을 지움"이라는 1줄을 (1) PR 본문에, (2) 코드 주석(예: `package.json` 위 주석 또는 첫 import 위)에 동시에 박는다. 1년 뒤 누구든 그 1줄로 채택 의도를 복원할 수 있도록.
- **확인 방법**: 의존성 추가 커밋 100%에 매핑 코멘트가 코드에 남아 있음.
- **첫 1주 권장 횟수**: **매 도구 추가 시** (없으면 0회).

## 3. 매주 단위 (Weekly 회고/학습)

### 액션 W1. 주말에 "어겼던 룰 1개 + 통증 분해 노트 1편 다듬기" 회고

- **연결되는 룰/패턴**: Rule 우선순위 (1, 9, 6 중심), Rule 10.
- **방법**: 주말 30분. (1) `20-abstract-rules.md`를 다시 읽고 이번 주 자신의 PR/페어 중 어느 룰을 어겼는지 1개를 짚어 "상황·이유·다음엔 어떻게" 3줄. (2) 한 주 동안 D1으로 모인 통증 분해 노트 중 가장 큰 1개를 골라 마무리한다 (단계·메커니즘·측정 3열을 채워 넣는다).
- **확인 방법**: 주 1편의 룰 회고 노트 + 주 1편의 다듬어진 통증 분해 노트.
- **첫 1주 권장 횟수**: **주 1회** (가장 빠뜨리지 말 것 — 누적의 입구).

### 액션 W2. 한 주의 익숙한 가정 1개를 종이에 적고 "깨질 시나리오 1개"를 떠올리기

- **연결되는 룰/패턴**: Rule 2, 문제 정의 패턴 3 (Assumption Audit).
- **방법**: 한 주 코드/회의에서 "당연히 X일 거예요"·"늘 그래왔어요"라고 넘긴 가정을 1개 골라 적는다. 그 가정이 깨질 시나리오를 1개 떠올려 같은 노트에 적는다 (사용자 행동, 플랫폼 동작, 매체 차이 중 하나에서).
- **확인 방법**: 주 1편의 "가정 + 깨질 시나리오" 노트.
- **첫 1주 권장 횟수**: **주 1회**.

---

## 4. 매월 단위 (Monthly)

### 액션 M1. 한 달 분 통증 분해 노트를 모아 "반복 메커니즘 1개" 패턴화

- **연결되는 룰/패턴**: Rule 10, 30-problem-definition-patterns.md 흐름도.
- **방법**: 월말 한 시간. D1·W1으로 쌓은 4~5장의 통증 분해 노트를 펼쳐, **반복되는 메커니즘**이 있는지 본다. 반복이 보이면 "내 영역에서의 추상화·도구화 신호"로 메모.
- **확인 방법**: 월 1편의 "이번 달 반복 패턴" 메모. 이게 자신만의 후속 학습 주제가 됨.
- **첫 1주 권장 횟수**: **월 1회**.

### 액션 M2. 자신의 현재 큰 작업에 대한 "한 문장 정렬축" 다듬기

- **연결되는 룰/패턴**: Rule 4, 해결 패턴 4 (Single-Sentence Alignment Axis).
- **방법**: 본인이 지금 하고 있는 가장 큰 작업(미션·프로젝트·학습 주제) 하나에 대해, **한 문장의 정렬축**을 적는다. "X와 비교해서 Y하게" 같이 비교 대상이나 매체 명시가 들어간 형태로. 추상적이면 다듬는다.
- **확인 방법**: 월 1개의 자기만의 정렬축 문장. 다음 달 결정에서 이 문장을 실제로 꺼내 봤는지 W1에서 점검.
- **첫 1주 권장 횟수**: **월 1회**.

## 5. 우테코 맥락 특화 액션

### 액션 U1. 페어 시작 전 "사용 코드 3줄"을 둘이 합의해서 먼저 적기

- **연결되는 룰/패턴**: Rule 6, 해결 패턴 2.
- **방법**: 페어로 새 기능을 시작할 때 키보드를 잡기 전, "이 기능을 어떻게 부를지" 3줄을 둘이 합의해 적는다. 시그니처가 어색하면 둘이 다듬는다 — 다듬은 결과가 첫 30분의 정렬축이 된다.
- **왜 우테코에서 효과적인가**: 페어 자체가 D2(인터페이스 우선)를 강제하는 가장 자연스러운 시점. 페어 변경 시에도 합의된 3줄이 인계 자료가 된다.
- **확인 방법**: 페어 세션마다 "합의된 사용 코드 3줄" 노트 1개.
- **첫 1주 권장 횟수**: **매 페어 시작 시**.

### 액션 U2. 미션 회고 양식에 "메커니즘 분해 1개 + 다음 라운드 1개" 칸을 의무로 추가하기

- **연결되는 룰/패턴**: Rule 1, Rule 9, 문제 정의 패턴 2 + 6.
- **방법**: 미션이 끝날 때 작성하는 회고에 두 칸을 의무로 둔다 — (1) **이번 미션의 가장 큰 통증을 메커니즘 분해**, (2) **이번 해결책이 만든 다음 라운드 통증 1개**. 두 칸이 비어 있으면 회고 미완.
- **왜 우테코에서 효과적인가**: 회고는 우테코 크루의 가장 일관된 산출물 — 회고 양식에 칸을 박으면 매 미션마다 룰 1·9가 강제 적용된다.
- **확인 방법**: 모든 미션 회고가 두 칸을 채운 상태로 제출.
- **첫 1주 권장 횟수**: **매 미션 종료 시**.

### 액션 U3. 크루 코드 리뷰에 "통증 매핑 질문 1개" 코멘트 룰

- **연결되는 룰/패턴**: Rule 7, 해결 패턴 3.
- **방법**: 새 도구·패턴을 도입한 PR을 리뷰할 때 **"이 변경이 어떤 통증의 어떤 메커니즘을 지우나요?"**라는 한 줄 질문을 코멘트로 단다. 리뷰어 한 사람이 매 PR마다.
- **왜 우테코에서 효과적인가**: 크루 리뷰는 횡적 학습의 자리 — 한 사람의 리뷰 코멘트가 다른 크루의 룰 7 체화로 전파된다. 본인이 PR 작성자일 때도 P4를 자연스럽게 들이게 됨.
- **확인 방법**: 도구 도입 PR 리뷰에 매핑 질문 1개 이상.
- **첫 1주 권장 횟수**: **도구 도입 PR을 볼 때마다**.

### 액션 U4. 페어 종료 시 "오늘의 안티 액션 1개"를 둘이 짚어 노트에 남기기

- **연결되는 룰/패턴**: 안티 액션 §6 전체.
- **방법**: 페어 세션 마지막 5분에 "오늘 우리가 §6 안티 액션 중 하나를 했던 부분 1개"를 둘이 짚어 노트에 남긴다. 안티 액션을 외부 시선으로 보는 것이 핵심 — 자기 혼자 짚기 어려운 자의적 판단을 페어가 보여 준다.
- **왜 우테코에서 효과적인가**: 페어가 자의적 판단을 외부 시선으로 보게 해 주는 가장 강력한 장치. 안티 액션을 페어 종료 의식으로 만들면 매 세션마다 점검됨.
- **확인 방법**: 매 페어 노트 마지막에 "오늘의 안티 액션 1개" 줄.
- **첫 1주 권장 횟수**: **매 페어 종료 시**.

## 6. 안티 액션 (피해야 할 것)

세 발표에서 발표자들이 명시적으로 비판한 행위를 일상 동작 단위로 번역한 STOP 사인:

### A1. "코드가 커져서 분리한다"는 PR 사유 금지

이름이 한 줄로 안 나오는 분리는 일단 멈추고, **이유를 한 줄 적어 본다**. 안 나오면 분리 자체를 취소. (E §4 액션 2 — `Item`·`popUpJungItem` 같은 의미 없는 단위가 늘어나는 것의 원인이 "적당히")

### A2. "X가 좋대서 도입했다" 도구 채택 금지

메커니즘 매핑 1줄(P4)이 없는 의존성 추가는 **PR 거절 사유**. (K1 §3 — SSR 채택 근거가 트렌드가 아닌 메커니즘 매핑이어야 한다는 명시적 거부)

### A3. "기능은 됐어요"로 마이그·기능 종료 선언 금지

**메트릭 동일성 + 운영 부담 + 다음 라운드 통증** 한 줄(P2)을 적기 전에는 "끝"이 아니다. (K1 §5-2, §5-3 — 안드/iOS 로깅 기준 차이로 데이터 맞추는 시간이 개발 시간보다 길었다)

### A4. "이건 당연히 X일 거예요"의 가정 위에서 결정 금지

가정 위에 결정을 쌓기 전에 **그 가정이 깨질 시나리오 1개**(W2)를 종이에 적는다. (K1 §5-4 — "배포만 하면 자동 전파"를 iOS 백그라운드 패턴이 깬 사례)

### A5. 같은 회의·합의를 도구화 없이 두 번 이상 반복 금지

같은 회의가 두 번째로 잡히는 순간 **"지금 우리가 다루는 정보를 데이터로 표현할 수 있는가?"** 한 줄을 적는다. 답이 "예"면 다음 회의 대신 도구화 PR을 연다. (K2 §1 — 회의가 반복된다는 것은 추상화 부재의 신호)
