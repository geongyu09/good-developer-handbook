# 좋은 개발자 패턴 추출 진행 절차

> **이 문서의 목적**
> 개발자 컨퍼런스 발표 스크립트 분석 문서들(`*.md`)을 바탕으로
> (1) 좋은 개발자가 되기 위한 추상 룰
> (2) 문제 정의 / 지혜로운 해결의 패턴
> (3) 우테코 크루·개발자 지망생용 액션 플랜
> 까지 단계적으로 도출하기 위한 작업 매뉴얼.
>
> **사용법**
> 각 Phase의 "Claude에게 보낼 프롬프트" 블록을 그대로 복사해 새 세션에 던지면 한 단계가 진행된다.
> 한 Phase가 끝나면 산출물 파일이 생기고, 다음 Phase가 그것을 입력으로 사용한다.

## Phase 0 — 입력 자료 점검

**목표:** 분석에 쓸 원본 문서가 충분한지, 분석 단위가 일관되는지 확인.

**완료 기준 (Definition of Done)**
- 분석 대상 `.md` 파일 목록이 확정됨
- 각 문서가 "문제 정의" / "해결" 구조를 갖고 있는지 한 줄 메모로 기록됨
- 부족하면 추가 자료 수집을 사용자에게 요청

### Claude에게 보낼 프롬프트

```
script/ 디렉터리의 .md 파일 중 00-procedure.md를 제외한 모든 파일을 점검해줘.
각 파일에 대해 다음을 한 줄씩 정리한 표를 만들어 출력만 해. (파일은 만들지 마)

| 파일명 | 발표자/출처 | 다루는 주제 한 줄 | "문제 정의" 섹션 유무 | "해결/원칙" 섹션 유무 | 메타 분석(발표자의 사고 패턴) 유무 |

마지막에:
- 분석에 충분한지 한 줄 평가
- 더 필요하면 어떤 종류의 자료를 추가하면 좋을지 1~2개 제안
```

## Phase 1 — 문서별 패턴 정규화 추출

**목표:** 각 원본 문서에서 동일한 스키마로 "문제 정의 패턴", "해결 패턴", "메타 인사이트"를 뽑아낸다. 이후 모든 단계는 이 정규화된 산출물만 본다.

**산출물:** `analysis/<원본파일명>.analysis.md` (원본 1개당 1개)

**정규화 스키마**
- `1. 한 줄 요약` — 발표가 풀려고 한 진짜 문제
- `2. 문제 정의 패턴` — 발표자가 어떻게 문제를 좁혀 들어갔는지 (단계별)
- `3. 해결 패턴` — 어떤 사고/원칙으로 해결했는지 (단계별)
- `4. 사용된 메타 도구` — 프레임 전환, 자기 코드 비판, 데이터 흐름 분리 등 재사용 가능한 사고 도구
- `5. 핵심 한 줄` — 다른 맥락에서도 인용 가능한 명제

### Claude에게 보낼 프롬프트

```
script/00-procedure.md의 Phase 1 섹션을 읽고 그 지침대로 작업해줘.

대상: script/ 디렉터리의 분석용 원본 .md 파일들
(00-procedure.md, analysis/ 하위 파일은 제외)

작업:
1. script/analysis/ 디렉터리를 생성
2. 원본 파일 1개당 analysis/<원본파일명>.analysis.md 1개를 생성
3. 각 산출물은 다음 5개 섹션을 정확히 같은 제목으로 가져야 함
   - ## 1. 한 줄 요약
   - ## 2. 문제 정의 패턴 (단계별, 발표자가 실제로 한 행위 중심)
   - ## 3. 해결 패턴 (단계별, 어떤 원칙·도구·사고를 적용했나)
   - ## 4. 사용된 메타 도구 (불릿 리스트, 각 항목은 "도구명 — 짧은 설명")
   - ## 5. 핵심 한 줄

규칙:
- 원본에 없는 내용을 창작하지 말 것. 원본에 있는 표현을 인용하거나 응축할 것.
- 각 패턴 단계마다 "(원본 §섹션 번호)" 식으로 어디서 왔는지 표시.
- 메타 도구 항목은 발표 도메인(컴포넌트, 웹뷰 등)을 걷어내고 일반화한 형태로 적을 것.
  예: "달력 데이터 추상화" → "변하지 않는 것과 변하는 것의 분리"

작업이 끝나면 생성된 파일 목록과 각 파일의 "5. 핵심 한 줄"만 모아서 보고.
```

## Phase 2 — 패턴 수렴 (Cross-Document)

**목표:** 여러 문서에서 반복되는 패턴은 묶고, 1개 문서에서만 등장하는 고유 패턴은 따로 모은다. 중복을 줄여 다음 단계의 입력을 좁힌다.

**산출물:** `10-pattern-convergence.md`

### Claude에게 보낼 프롬프트

```
script/00-procedure.md의 Phase 2를 읽고 진행해줘.

입력: script/analysis/*.analysis.md 의 모든 파일
출력: script/10-pattern-convergence.md (1개)

산출물 구조:

# 패턴 수렴 분석

## A. 문제 정의 단계에서 반복되는 패턴
표 형식:
| 패턴 이름 | 핵심 동작 | 등장한 문서 (≥2개) | 대표 인용 |

## B. 해결 단계에서 반복되는 패턴
표 형식:
| 패턴 이름 | 핵심 동작 | 등장한 문서 (≥2개) | 대표 인용 |

## C. 1개 문서에만 등장한 고유 패턴 (보존 가치 있는 것만)
- 패턴 이름 — 출처 — 왜 살릴 가치가 있는가

## D. 패턴 간 충돌/긴장 관계
- 한 패턴이 다른 패턴과 트레이드오프 관계인 경우 짚어줄 것
  (예: "표준 인터페이스 추구" vs "도메인 특화 표현력")

## E. 한 줄 결론
- 세 발표를 관통하는 단 한 줄

규칙:
- "비슷해 보이는 것"을 무리하게 묶지 말 것. 동작 메커니즘이 같을 때만 같은 패턴.
- 인용은 짧게(1~2문장), 출처 표시 필수: (파일명 §섹션)
```

## Phase 3 — 추상 룰 도출 (목표 1)

**목표:** 수렴된 패턴에서 도메인을 모두 걷어낸 "좋은 개발자가 되기 위한 추상 룰" 5~10개를 뽑는다.

**산출물:** `20-abstract-rules.md`

**룰의 품질 기준**
- 도메인 단어(컴포넌트, 웹뷰 등)가 룰 본문에 안 들어와야 함
- "왜 그 룰이 좋은가"의 근거가 1줄 이상 있어야 함
- 룰을 어겼을 때 어떤 통증이 생기는지 1줄 이상 있어야 함

### Claude에게 보낼 프롬프트

```
script/00-procedure.md의 Phase 3을 읽고 진행해줘.

입력: script/10-pattern-convergence.md
출력: script/20-abstract-rules.md

산출물 구조:

# 좋은 개발자가 되기 위한 추상 룰

각 룰은 다음 형식으로 작성:

## Rule N. <한 줄 명제>
- **왜 좋은가**: (1~2줄)
- **어겼을 때의 통증**: (1~2줄)
- **근거 패턴**: (Phase 2 산출물의 어떤 패턴에서 도출됐는지)
- **반례/한계**: (이 룰이 적용되지 않거나 위험한 경우)

요건:
- 룰 개수는 5~10개
- 룰 본문에 컴포넌트/웹뷰/브릿지 등 특정 도메인 단어 금지 (근거 패턴 칸에서는 OK)
- "코드를 잘 짜라" 같은 동어반복 금지. 행위가 명시되어야 함.
- 룰 사이 중복 없을 것
- 마지막에 "## 룰 우선순위" 섹션을 두고, 가장 먼저 체화해야 할 1~3개를 골라 이유와 함께 정렬
```

## Phase 4 — 문제 정의 패턴 카탈로그 (목표 2-a)

**목표:** "잘하는 개발자는 문제를 어떻게 올바르게 정의하는가"의 패턴을 재사용 가능한 카탈로그로 만든다.

```

# 문제를 올바르게 정의하는 패턴 카탈로그

## 패턴 N. <패턴 이름>
- **언제 쓰는가**: (어떤 상황에서 이 패턴을 꺼내야 하는지)
- **무엇을 하는가**: (구체적 행위, 2~4단계로)
- **출처 사례**: (어느 발표의 어느 장면에서 이 패턴이 나왔는지 1~2줄)
- **이 패턴이 막아주는 실패**: (이 패턴을 안 쓰면 어떤 잘못된 정의가 나오는지)
- **자가 점검 질문**: (1~3개. 이 질문에 답해보면 패턴을 적용했는지 알 수 있는)

요건:
- 패턴 5~8개
- 단순 나열이 아니라, "추상→구체" 또는 "통증→메커니즘 분해→재정의" 같은 흐름이 보이도록 정렬
- 마지막에 "## 흐름도" 섹션: 패턴들을 어떤 순서로 적용하면 한 사이클이 되는지 짧게 정리 (5줄 이내)
```

## Phase 5 — 지혜로운 해결 패턴 카탈로그 (목표 2-b)

**목표:** "잘하는 개발자는 어떻게 지혜롭게 해결하는가"의 패턴을 재사용 가능한 카탈로그로 만든다.

**산출물:** `31-wise-solution-patterns.md`

**"지혜롭다"의 조작적 정의**
- 더 작은 결정으로 더 많은 변화를 흡수
- 자의적 판단을 측정 가능한 기준으로 바꿈
- 단기 통증을 감수하더라도 시스템의 본질을 단순화함
- 협업 비용·인지 비용을 코드 구조로 흡수함

### Claude에게 보낼 프롬프트

```
script/00-procedure.md의 Phase 5를 읽고 진행해줘.

입력:
- script/10-pattern-convergence.md (특히 B, D 섹션)
- script/analysis/*.analysis.md 의 "3. 해결 패턴", "4. 사용된 메타 도구"

출력: script/31-wise-solution-patterns.md

먼저 procedure 문서의 "지혜롭다의 조작적 정의" 4가지를 그대로 도입부에 옮겨 적은 뒤 시작.

각 패턴 형식:

## 패턴 N. <패턴 이름>
- **핵심 메커니즘**: (왜 이 해결이 "지혜로운가" — 위 4가지 정의 중 어디에 해당하는지 명시)
- **적용 절차**: (3~5단계)
- **출처 사례**: (어느 발표에서 어떻게 쓰였는지)
- **흔한 오용**: (잘못 적용하면 어떻게 망가지는지)
- **함께 쓰면 좋은 패턴**: (목표 2-a 카탈로그의 어떤 문제 정의 패턴과 짝을 이루는지)

요건:
- 패턴 5~8개
- "분리하라", "추상화하라" 같은 추상 명령이 아니라, **무엇을 무엇으로부터 분리하는지**까지 적을 것
- 마지막에 "## 안티패턴 짧은 목록" — 이 발표들에서 발표자가 직접 "이렇게 하면 안 된다"고 짚은 것들
```

## Phase 6 — 액션 플랜 도출 (목표 3)

**목표:** 추상 룰과 두 카탈로그를 입력으로, 우테코 크루·개발자 지망생이 **이번 주부터** 실천할 수 있는 액션 플랜을 만든다.

**산출물:** `40-action-plan.md`

**액션의 품질 기준**
- 외부 환경 변화 없이 본인만으로 시작할 수 있어야 함
- 결과를 직접 확인할 수 있어야 함 (코드, 노트, 회고록 등 산출물 명시)
- 시간 단위가 명시 (한 번 / 매일 / 매주 / 매 PR / 매 코드 리뷰 등)

### Claude에게 보낼 프롬프트

```
script/00-procedure.md의 Phase 6을 읽고 진행해줘.

입력:
- script/20-abstract-rules.md
- script/30-problem-definition-patterns.md
- script/31-wise-solution-patterns.md

출력: script/40-action-plan.md

산출물 구조:

# 우테코 크루·개발자 지망생을 위한 액션 플랜

## 0. 이 액션 플랜의 사용법
(3~5줄. 무리하게 한꺼번에 적용하지 말고 한 사이클에 1~2개만 시도하라는 등의 지침)

## 1. 매일 단위 (Daily)
- 액션 N. <행위>
  - **연결되는 룰/패턴**: (Phase 3/4/5의 어떤 항목과 매칭되는가)
  - **방법**: (구체적, 따라할 수 있게)
  - **확인 방법**: (오늘 이걸 했는지 어떻게 아는가 — 산출물·표지)
  - **첫 1주 권장 횟수**

## 2. 매 PR/매 과제 단위
(같은 형식)

## 3. 매주 단위 (Weekly 회고/학습)
(같은 형식)

## 4. 매월 단위 (Monthly)
(같은 형식)

## 5. 우테코 맥락 특화 액션
- 페어 프로그래밍, 미션 회고, 크루 코드 리뷰 같은 우테코 고유 활동에서만 가능한 액션 별도 분리
- 각 액션이 "왜 우테코에서 특히 효과적인가" 1줄 근거
```
