# Effective Component: 지속 가능한 성장과 컴포넌트

> **출처:** 토스 SLASH 22 — 한재엽 (토스페이먼츠 프런트엔드 개발자)
> **링크:** https://www.youtube.com/watch?v=fR8tsJ2r7Eg

복잡한 결제 대시보드 제품을 만들면서 마주한 "변경에 유연한 컴포넌트는 어떻게 만들 수 있는가"라는 질문을, 발표자가 어떤 흐름으로 정의하고 해결했는지 정리한 문서입니다.

---

## 1. 문제 인식 — 현상에서 출발하기

발표자는 추상적인 원칙이 아니라, **개발자가 실제로 마주하는 감정적인 장면**에서 시작합니다.

- "이번 스프린트에 변경이 발생했어요" — 방금 구현한 기능을 다시 손대야 하는 상황
- 작업한 게 아깝고, 아쉬운 감정이 든다
- 그런데 이런 변경은 **예외가 아니라 일상**

> 핵심 인사이트: 변경이 많은 이유는 단순하다. **제품이 변경되지 않아도 되는 조건은 단 하나 — 사용자가 잘 사용하고 있을 때.** 그렇지 않다면 디자인도, API도, 제품 방향 자체도 언제든 바뀐다.

→ 변경은 **나쁜 것이 아니라 성장의 신호**라는 관점 전환. 놓쳤던 고객 니즈를 발견한 것이고, 곧 기회다.

---

## 2. 문제 정의 — "예측"에서 "대응"으로 프레임 바꾸기

### 첫 번째 프레이밍

> 변경은 미리 알 수 없다. 그렇다면 **예측하지 말고 대응하자.**

이 한 줄이 발표 전체의 방향을 잡습니다. "어떤 변경이 올지 맞히는 설계"가 아니라 "어떤 변경이 와도 흡수할 수 있는 설계"를 만드는 것이 목표입니다.

### 문제를 더 뾰족하게 좁히기

발표자는 곧바로 해결책을 던지지 않고, **문제를 한 번 더 좁혀들어갑니다.**

1. 변경에 대응하기 힘들었던 경험을 떠올린다
2. 왜 힘들었는지 들여다본다
3. 본인이 3년 전에 짠 코드를 직접 사례로 든다 — `Item`, `popUpJung Item`처럼 의미를 알 수 없는 네이밍
4. "이 코드를 변경해야 한다고 하면? 어디부터 봐야 할지 모른다" — **변경 비용 = 탐색 비용 + 영향 범위 추적 비용**이라는 진짜 통증을 짚어낸다
5. 원인 추적: "만들다 보니 페이지가 커졌고, **적당히** 덜어내면서 이렇게 됐다"
6. **"적당히"라는 행위를 일반화할 수 있는 기준은 없을까?** → 이게 진짜 풀어야 할 문제

> 좋은 문제 정의의 패턴:
> - 추상적인 불편(`코드가 복잡하다`)에서 시작
> - 본인의 실패 코드를 증거로 제시
> - 통증의 메커니즘을 분해(`어디 수정할지 모름 → 전체를 봐야 함`)
> - 행위의 임의성(`적당히`)을 문제의 핵심으로 지목

## 3. 해결 전략 — 세 가지 원칙

발표자는 "변경에 제어 가능한 단위 = 컴포넌트"로 범위를 한정한 뒤, 변경에 유연한 컴포넌트의 특징을 세 가지로 정리합니다.

### 원칙 1. 헤드리스 기반의 추상화

컴포넌트의 세 역할 — **데이터 관리 / UI 표현 / 상호작용** — 을 분리합니다.

**예시 1: 달력 컴포넌트**
- 달력 데이터 자체(2×2 배열의 Date 객체)는 거의 변하지 않는다
- 달력의 UI는 디자인이 바뀔 때마다 변한다
- → `useCalendar` 훅으로 **데이터를 추상화**, UI는 데이터를 받아 그리기만 함
- 다른 디자인의 달력이 필요해도 `useCalendar`는 재사용 가능

**예시 2: 버튼의 long press**
- 버튼 내부에 long press 로직이 들어가면 컴포넌트가 비대해짐
- → `useLongPress` 훅으로 **상호작용을 추상화**, 버튼은 보여주기에만 집중
- 다른 컴포넌트에서도 long press가 필요하면 그대로 가져다 사용

> 키 메시지: 헤드리스 패턴은 **한 가지 문제에만 집중하기 때문에 더 많은 곳에서 사용 가능하고, 다른 변경으로부터 격리된다.**

### 원칙 2. 한 가지 역할만 하는 컴포넌트의 조합

복잡한 컴포넌트일수록, 작은 컴포넌트의 **합성(composition)** 으로 만들어야 합니다.

**예시: Select UI**
- 처음에는 `<Select>` 하나에 라벨, 인풋, 옵션, 메뉴 표시 여부가 다 들어가 있음
- 라벨만 바뀌어도 재사용 불가, 인풋 버튼만 다른 걸 쓰고 싶어도 불가
- 분해:
  - `Dropdown` — 메뉴 열림/닫힘 상태 관리
  - `DropdownTrigger` — 상태를 토글하는 상호작용
  - `DropdownMenu` — 옵션 영역(상태에 따라 노출)
  - `DropdownItem` — 각 항목과 선택 이벤트
- 결과: `Select` 컴포넌트와 트리거로 들어가는 `InputButton`은 **서로의 존재를 모른다.** 트리거가 바뀌어도 `Select`는 그대로.

**더 복잡한 사례: 모달 + 체크박스 다중 선택**
- UI는 복잡해 보이지만, **데이터 흐름으로 보면 Select와 동일** ("옵션을 제공하고 선택한 값을 보여준다")
- 다른 점은 "여러 개 선택 가능"뿐
- → 같은 `Dropdown` 컴포넌트로 구성, 메뉴 자리에 `DropdownModal`을 끼워 넣고, 적용 버튼을 form submit으로 처리

> 키 메시지: **데이터 흐름이 같으면 인터페이스도 같게 구성할 수 있다.** UI 모양에 속지 말고 데이터 흐름을 보라.

### 원칙 3. 도메인을 포함하는 컴포넌트와 그렇지 않은 컴포넌트의 분리

**`FrameworkSelect`처럼 도메인이 박힌 이름** vs **`MultiSelect`처럼 일반적인 이름**.

- 사람은 **기존 배경 지식으로 새것을 해석**한다
- 따라서 컴포넌트 인터페이스는 **표준(HTML `<select>` 등)에 가까울수록** 더 많은 사람이 빠르게 이해한다
- props 네이밍에서 도메인 맥락을 걷어내면, 자연스럽게 일반적인 이름으로 수렴된다
- 도메인을 모르는 컴포넌트(`MultiSelect`)와 도메인을 아는 컴포넌트(`FrameworkSelect`)를 **레이어로 구분**한다
- 도메인 컴포넌트는 비즈니스 로직을 직접 처리하되, **UI 로직은 일반 컴포넌트에 위임**한다

> 키 메시지: **인터페이스가 표준에 가까울수록 이해 비용이 낮아진다.** 일반화는 단순히 미적 취향이 아니라 인지 부담을 줄이는 도구.

## 4. 실천 방법 — 두 가지 액션 아이템

원칙을 알아도 코드를 짜는 순간엔 다시 "적당히" 모드로 돌아가기 쉽습니다. 발표자는 두 가지 습관을 제안합니다.

### 액션 1. 인터페이스를 먼저 고민하기

> 구현해야 하는 기능이 **이미 만들어져 있다고 가정하고**, 그것을 **사용하듯이** 코드를 작성해본다.

- 예: 좌측 은행 목록 + 우측 폼 화면 → "이건 이미 열려 있는 Select UI다"라고 보면, 좌측 = 메뉴, 우측 = 선택된 값의 표시
- 사용 코드부터 쓰면, 컴포넌트가 어떤 데이터를 다루고 어떤 인터페이스를 가져야 하는지가 명확해짐
- **"무엇을 받고 무엇을 내보낼지가 어떻게 구현할지보다 먼저 결정되어야 한다."**

### 액션 2. 컴포넌트를 나누기 전에 "왜 나누는가" 다시 묻기

- 복잡도를 낮추기 위해 나누는 것인지
- 재사용을 위해 나누는 것인지
- **꼭 분리해야 하는 컴포넌트인지**
- 이유 없이 "크다"는 이유로 나누면, 결국 `Item`, `popUpJungItem` 같은 의미 없는 컴포넌트가 생긴다

## 5. 발표자가 사용한 문제 해결 방식 (메타 분석)

이 발표 자체가 좋은 문제 정의 → 해결의 모범 사례입니다. 패턴을 뽑으면:

| 단계 | 발표자가 한 일 | 일반화 |
|---|---|---|
| 1. 통증 포착 | "스프린트 중 변경이 왔다"는 감정적 장면 | 누구나 공감하는 통증에서 출발 |
| 2. 통증 일반화 | "변경은 일상이고, 예측 불가하다" | 통증이 일회성이 아니라 구조적임을 보임 |
| 3. 프레임 전환 | "예측하지 말고 대응하자" | 풀 수 없는 문제를 풀 수 있는 문제로 재정의 |
| 4. 자기 코드로 증거 제시 | 3년 전 본인 코드의 실패를 공개 | 추상론이 아닌 구체로 신뢰 확보 |
| 5. 통증의 원인 분해 | "어디 수정할지 모름 = 탐색 비용" | 통증 = X + Y + Z 로 쪼개 측정 가능하게 |
| 6. 임의성을 문제 핵심으로 지목 | "적당히 나눈다"의 "적당히"를 공격 대상으로 삼음 | 자의적 판단 = 일반화의 적 |
| 7. 원칙 도출 | 3가지 특성 (헤드리스 / 합성 / 도메인 분리) | 무엇이 좋은지 측정 가능한 기준 제시 |
| 8. 예제로 검증 | 달력, 버튼, Select, 모달, 은행 목록 | 원칙이 다양한 상황에서 동일하게 작동함을 보임 |
| 9. 실천 도구화 | 두 가지 액션 아이템 | 원칙이 일상 습관으로 내려오게 만듦 |

---

## 6. 가져갈 한 줄

> **변경을 예측하지 말고, 변경에 대응하라.**
> 그 대응의 단위는 **헤드리스로 추상화되고, 한 가지 일만 하며, 도메인이 명확히 분리된 컴포넌트**다.
> 그리고 컴포넌트를 짜기 전에 **인터페이스를 먼저, 분리 이유를 다시 한 번** 묻자.
